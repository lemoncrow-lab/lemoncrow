[assistant] Here's a multi-stage `Dockerfile` optimized for a small final image:

```dockerfile
# syntax=docker/dockerfile:1

# ---- Stage 1: Build ----
FROM node:22-alpine AS build
WORKDIR /app
COPY package.json package-lock.json ./
RUN npm ci
COPY . .
RUN npm run build

# ---- Stage 2: Production dependencies ----
FROM node:22-alpine AS deps
WORKDIR /app
COPY package.json package-lock.json ./
RUN npm ci --omit=dev

# ---- Stage 3: Runtime ----
FROM node:22-alpine AS runtime
ENV NODE_ENV=production
WORKDIR /app
USER node
COPY --chown=node:node --from=deps  /app/node_modules ./node_modules
COPY --chown=node:node --from=build /app/dist        ./dist
COPY --chown=node:node package.json ./
EXPOSE 3000
CMD ["node", "dist/index.js"]
```

## Why it stays small

- **Three stages, one shipped.** Only the final `runtime` stage becomes your image. The TypeScript compiler, source files, and all `devDependencies` live in the `build`/`deps` stages and are discarded.
- **Separate `deps` stage** runs `npm ci --omit=dev`, so `node_modules` in the final image contains *only* production dependencies — no `typescript`, `@types/*`, linters, etc.
- **`alpine` base** (~50 MB vs ~380 MB for `node:22`). Swap for `node:22-slim` if you hit musl/native-module issues (e.g. some `bcrypt`, `sharp`, or `prisma` builds).
- **Layer caching:** copying `package*.json` and installing before copying source means `npm ci` re-runs only when dependencies change, not on every code edit.

## Things to adjust for your app

1. **Build output path** — this assumes `tsconfig.json` emits to `./dist` and your entrypoint is `dist/index.js`. Match `outDir` and the `CMD`.
2. **`npm run build`** — assumes a `"build": "tsc"` (or similar) script exists in `package.json`.
3. **Port** — change `EXPOSE 3000` to your actual port.
4. **`.dockerignore`** — add one to keep `node_modules`, `.git`, and `dist` out of the build context:

```
node_modules
dist
.git
npm-debug.log
```
