## The bug: seconds vs. milliseconds

JWT's `exp` is a **Unix timestamp in seconds** (per RFC 7519). `Date.now()` returns **milliseconds**. They're off by a factor of 1000, so any raw comparison between them is wrong.

`Date.now()` (~`1.75e12`) is always ~1000× larger than `exp` (~`1.75e9`). So depending on how your check is written, the expiry branch either never fires or always fires — and in the common "valid when `exp > Date.now()`" / "expired when `Date.now() > exp`" shapes, real expiry is never detected correctly and expired tokens slip through.

## Fix — put both sides in the same unit

```js
// expired when now >= exp; convert exp (s) to ms
if (Date.now() >= decoded.exp * 1000) {
  return res.status(401).json({ error: 'Token expired' });
}

// equivalently, convert now to seconds:
if (Math.floor(Date.now() / 1000) >= decoded.exp) { /* expired */ }
```

Use `>=` for the expired condition, not `<`.

## Better fix — don't hand-roll it

A manual `Date.now()`/`exp` check usually means you're on `jwt.decode()`, which verifies **nothing** — not expiry *and not the signature*, so forged tokens also get through. Let `jwt.verify()` do it; it validates `exp` in the correct unit and throws `TokenExpiredError`:

```js
try {
  const decoded = jwt.verify(token, SECRET); // checks signature + exp
  req.user = decoded;
  next();
} catch (err) {
  return res.status(401).json({ error: 'Invalid or expired token' });
}
```

Add `{ clockTolerance: 5 }` (seconds) to `verify` if you need leeway for clock skew.

## Two adjacent traps to watch

- **Missing `exp`:** `undefined * 1000` is `NaN`, and `Date.now() >= NaN` is `false` — a token with no `exp` sails through. Reject when `exp` is absent. (`jwt.verify` handles this for you.)
- **Comparison direction:** confirm the operator actually rejects on expiry rather than admitting it.
