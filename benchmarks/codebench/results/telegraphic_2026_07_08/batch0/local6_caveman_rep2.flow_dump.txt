[assistant] Before microservices, check these factors:

**1. Is monolith itself the problem?**
Most Django slowness NOT architecture. Profile first. Common wins:
- Missing DB indexes, N+1 queries (use `django-silk` / `nplusone`)
- No query caching, no `select_related`/`prefetch_related`
- Sync work that should be async (offload to Celery)
- Single process, no horizontal scaling behind load balancer

Microservices fix none of these. Slow query stay slow after split.

**2. Team + org shape (Conway's Law)**
Services map to teams. One team of 5 running 12 services = pain. Need clear ownership per service. If team can't own boundaries, don't split.

**3. Where the real bottleneck live?**
Split only hot part. If one module (e.g. reporting, image processing) eats CPU/memory, extract that one. Rest stay monolith. This "strangler" approach beat big-bang rewrite.

**4. Data coupling**
Hardest part. Monolith = one DB, easy JOINs + transactions. Split DB means:
- No cross-service JOIN
- Distributed transactions (sagas, eventual consistency)
- Data duplication + sync
If your tables tightly joined everywhere, split very costly.

**5. Operational cost added**
Microservices bring new burden:
- Service discovery, network calls (latency + failure modes)
- Distributed tracing, log aggregation
- CI/CD per service, versioning, deploy orchestration
- Auth between services
Need this infra + skill before, not after.

**6. Network latency replaces function calls**
In-process call = nanoseconds. Network call = milliseconds + can fail. Chatty services slower than monolith. Bad boundary = worse performance.

**7. Testing + debugging harder**
One stack trace becomes distributed trace across services. Local dev needs many services running.

## Recommend
Order of attack:
1. **Profile.** Find actual slow path with data, not guess.
2. **Optimize monolith** — indexes, caching, Celery, read replicas, more app servers. Usually enough.
3. **Modularize inside monolith** — clean app boundaries, no cross-app model reach-in. Cheap, reversible.
4. **Extract one service** only if one bounded context has different scaling/team needs.

Microservices solve org + independent-scaling problems, not raw speed. Team of good size hitting deploy-coupling or one component needing huge separate scale = valid reason. "App slow" alone = usually not.

What the actual symptom? Slow requests, slow deploys, or team stepping on each other? Answer change advice.
