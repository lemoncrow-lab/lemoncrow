[assistant] # Git Rebase vs. Git Merge

Both integrate changes from one branch into another, but they build **different histories**.

## Merge

`git merge` takes the commits from your branch and the target and ties them together with a new **merge commit** that has two parents.

```
      A---B---C  feature
     /         \
D---E---F---G---M  main
```

- History is **preserved exactly** as it happened — non-destructive.
- The merge commit records *when* and *where* branches came together.
- Downside: many merge commits can clutter history and make `git log` noisy.

## Rebase

`git rebase` **replays** your branch's commits on top of the target, creating new commits with new hashes.

```
D---E---F---G---A'---B'---C'  feature (rebased onto main)
```

- Produces a **linear, clean** history — as if you'd written the work starting from the latest main.
- No merge commits.
- Downside: it **rewrites history** (new commit SHAs), so it's destructive to the original commits.

## When to use each

**Use merge when:**
- Integrating a completed feature branch into a shared/long-lived branch (`main`, `develop`).
- You want an accurate, auditable record of when work was integrated.
- The branch is **public/shared** — others may have based work on it.

**Use rebase when:**
- Updating your *local, in-progress* feature branch with the latest `main` before finishing.
- Cleaning up your own commits before opening a PR (`git rebase -i` to squash, reword, reorder).
- You want a tidy, linear history that's easy to read and `git bisect`.

## The tradeoffs at a glance

| | Merge | Rebase |
|---|---|---|
| History | True, non-linear | Linear, curated |
| Commit hashes | Unchanged | Rewritten |
| Merge commits | Yes | No |
| Safe on shared branches | ✅ | ⚠️ Avoid |
| Conflict resolution | Once, in the merge commit | Potentially per replayed commit |

## The golden rule

> **Never rebase commits that others have already pulled.** Rewriting shared history forces everyone else into painful reconciliations.

Rebase *private* work to keep it clean; merge *public* work to keep it honest.

## A common hybrid workflow

1. Work on a feature branch.
2. Periodically `git rebase main` to stay current (while the branch is still yours alone).
3. When done, `git merge --no-ff feature` into main — often with a single merge commit that marks the feature's arrival.

This gives you clean, linear feature history *and* a clear integration point. Many teams also enable "squash and merge" on their PR platform, which collapses a branch into one commit on main — a middle ground between the two.
