[assistant] Before splitting anything, get honest about **why it's slow**, because most Django monolith slowness has nothing to do with being a monolith — and microservices won't fix it, they'll just distribute it.

## First: is "monolith" actually the problem?

Profile before you architect. Common culprits that survive a rewrite:
- **N+1 queries** — missing `select_related`/`prefetch_related`. Add `django-silk` or `nplusone` and look.
- **Missing DB indexes** / slow queries — check `pg_stat_statements`.
- **Synchronous work in the request path** — emails, PDF generation, third-party API calls that belong in Celery.
- **No caching layer** — Redis for hot reads, per-view or fragment caching.
- **Undersized DB / single connection pool** — often a config problem, not an architecture one.

If you haven't done these, do them first. They're days of work vs. quarters, and they're reversible. A well-tuned Django monolith serves very large traffic.

## Key factors if you're seriously considering the split

**1. Do you have clean domain boundaries?**
Can you draw lines where modules talk through narrow, stable interfaces? If your models are a tangle of cross-app foreign keys and shared `select_related` chains, you don't have boundaries yet — you have a distributed monolith waiting to happen. Enforce module boundaries *inside* the monolith first (e.g. `import-linter`). If you can't do it in one codebase, you can't do it across services.

**2. The data problem is the real problem.**
Splitting code is easy; splitting the database is not. Cross-service foreign keys become network calls. Transactions that were ACID become eventual consistency, sagas, and reconciliation jobs. Ask: which tables would each service own, and how many joins currently cross those lines? That number is your migration cost.

**3. Independent scaling need — is it real?**
Microservices pay off when *different parts scale differently* (e.g. a search or media pipeline hammered while the rest idles). If your whole app scales together, you can just run more monolith instances behind a load balancer — same benefit, none of the distributed-systems tax. You can also extract *one* hot component (the classic "strangler fig" approach) without going full microservices.

**4. Team topology.**
Conway's Law is real. Microservices are an *organizational* solution — they let independent teams deploy independently. One team of 6 running five services just inherits five deploy pipelines and no autonomy gain. How many teams do you have, and do they actually block each other on deploys today?

**5. Operational maturity — the tax you'll pay.**
Distributed systems need: centralized logging/tracing (you can no longer read one stack trace), service discovery, per-service CI/CD, network retry/timeout/circuit-breaker logic, distributed debugging. If you don't have solid observability on the monolith now, you'll be flying blind across ten services.

**6. Latency and failure modes.**
In-process calls become network calls: latency, partial failures, cascading timeouts. A page that did 8 function calls might now do 8 HTTP round-trips.

## Recommendation

1. **Profile and fix the obvious wins first** (queries, indexes, Celery, caching). Measure again.
2. If still bound, **modularize inside the monolith** — enforce boundaries, kill cross-app coupling.
3. If a specific component has a genuinely different scaling/team profile, **extract just that one** (strangler fig). Keep the rest as a modular monolith.

Full microservices is the right call for a minority of teams — usually ones with multiple independent teams hitting deploy contention, not ones with a slow database. Slowness alone almost never justifies it.
