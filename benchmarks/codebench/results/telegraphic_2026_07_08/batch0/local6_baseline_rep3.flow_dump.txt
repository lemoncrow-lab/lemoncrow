[assistant] Before you split anything, it's worth naming the trap: **"the monolith is slow" and "we should do microservices" are two unrelated statements.** Microservices don't make things faster — they usually add network latency, serialization overhead, and distributed-systems complexity. So the first factor is the most important one.

## 1. Is slowness even an architecture problem?

Most Django slowness is *not* solved by splitting the codebase. Rule these out first, because they're cheaper and often the actual cause:

- **N+1 queries** — the classic Django killer. Audit with `django-debug-toolbar` or `nplusone`; fix with `select_related`/`prefetch_related`.
- **Missing DB indexes** / slow queries — check `pg_stat_statements`.
- **No caching layer** — Redis/Memcached for expensive reads, per-view or fragment caching.
- **Synchronous work that should be async** — offload to Celery (email, PDF generation, third-party API calls).
- **Under-provisioned workers/DB** — sometimes it's just gunicorn worker count or DB connection pooling.

If you haven't profiled and exhausted these, **stop** — microservices will add complexity without fixing the root cause. Get real numbers (APM: Datadog, New Relic, Sentry Performance) showing *where* time goes.

## 2. Is the bottleneck actually a candidate for extraction?

Microservices help scaling when **one part of the system has fundamentally different resource needs** — e.g. a CPU-heavy image processor, an ML inference endpoint, or a high-throughput ingestion pipeline that you want to scale independently of the web tier. That's a legitimate reason. "The whole app is slow" is not — you'd just get many slow services.

## 3. Organizational readiness (Conway's Law)

Microservices are as much an org structure as a technical one. Ask:
- Do you have **multiple teams** that keep stepping on each other in the same codebase? Service boundaries can decouple teams. One team of 5? A monolith is almost certainly right.
- Can you staff the **operational overhead** — CI/CD per service, distributed tracing, on-call for N services, service discovery, infra-as-code?

## 4. Domain boundaries

Can you draw **clean seams** through the domain? If your models are a tangled web of foreign keys and cross-cutting queries, splitting produces distributed transactions and chatty inter-service calls — the worst of both worlds. Do a Domain-Driven Design exercise and find the bounded contexts *before* touching code. If you can't cleanly separate them in the monolith, you can't cleanly separate them into services.

## 5. Data ownership

This is where most migrations bleed. Each service should own its data:
- How do you handle a query that currently `JOIN`s across what would become two services?
- Are you ready for **eventual consistency** and **distributed transactions** (sagas, outbox pattern) instead of a single ACID transaction?
- Shared database between services is an anti-pattern that reintroduces the coupling you're trying to escape.

## 6. The cost side of the ledger

Splitting adds, permanently: network latency between calls, versioned API contracts, integration testing across services, harder local dev, distributed debugging, more infra spend. Be honest that you're trading one kind of complexity for another.

---

## A pragmatic middle path

Before full microservices, consider these — they capture much of the benefit at a fraction of the cost:

1. **Modular monolith** — enforce clean module boundaries *within* Django (separate apps, no cross-module ORM reaching, explicit internal APIs). This is often the real fix and makes any future extraction trivial.
2. **Extract one service** — the highest-value, cleanest-boundary piece (e.g. the CPU-heavy component from #2). Learn the operational cost on one service before committing to twenty.
3. **Scale the monolith first** — read replicas, more app servers behind a load balancer, Celery for async work. Django scales to very large traffic as a monolith.

**My recommendation:** profile first to prove where the time actually goes. Nine times out of ten the win is query optimization + caching + async offloading, and the microservices debate evaporates. If a genuine boundary with distinct scaling needs emerges, extract *that one piece* — don't big-bang the whole thing.
