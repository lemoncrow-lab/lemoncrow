[assistant] Database connection pooling is a technique for managing and reusing database connections instead of opening a new one for every request.

## The problem it solves

Opening a database connection is expensive. Each new connection involves:
- A TCP handshake (and often a TLS handshake)
- Authentication (username/password, credential checks)
- Session setup on the database server (memory allocation, backend process/thread creation)

This can take tens to hundreds of milliseconds. If your app opens and closes a connection for every query, that overhead dominates, and under load you can exhaust the database's connection limit or spawn thousands of backend processes.

## How pooling works

A **connection pool** maintains a set of open, ready-to-use connections:

1. On startup (or on demand), the pool opens a number of connections and keeps them alive.
2. When your code needs the database, it **borrows** a connection from the pool.
3. It runs its queries.
4. When done, it **returns** the connection to the pool instead of closing it — so the next request can reuse it.

If all connections are in use, new requests either wait for one to free up or fail after a timeout.

```
App threads → [ Pool: conn1, conn2, conn3, ... ] → Database
              borrow ↑            ↓ return
```

## Key configuration knobs

- **Min / max pool size** — how many connections to keep idle and the ceiling on total connections.
- **Acquire/connection timeout** — how long a caller waits for a free connection before erroring.
- **Idle timeout** — close connections unused for a while to release resources.
- **Max lifetime** — retire connections after some age to avoid stale/leaked server-side state and to rebalance across DB replicas.
- **Validation / health check** — test a connection (e.g. a lightweight ping) before handing it out, so you don't hand back a dead one.

## Benefits

- **Lower latency** — no per-request connection setup.
- **Higher throughput** — reuse amortizes the handshake cost.
- **Resource protection** — caps concurrent connections so you don't overwhelm the database.

## Things to watch out for

- **Sizing:** A bigger pool isn't always better. The database has a finite capacity (e.g. Postgres backends are relatively heavy). A common guideline is a pool of roughly `cores × 2` per app instance, tuned by testing — not hundreds of connections.
- **Total connections across instances:** With N app instances each holding a pool, the database sees `N × pool_size`. This is why heavily-scaled or serverless apps often add an external pooler like **PgBouncer** in front of Postgres.
- **Connection leaks:** If borrowed connections aren't returned (e.g. an exception path skips the release), the pool drains and everything blocks. Use try/finally or your framework's scoped/context-managed helpers.
- **Stale connections:** Network blips or DB restarts can leave dead connections in the pool; validation and max-lifetime settings mitigate this.
- **Transaction/session state:** A returned connection may retain session settings, temp tables, or an open transaction. Pools should reset connection state on return.

## Common implementations

- **Java:** HikariCP, Tomcat JDBC
- **.NET:** built into ADO.NET / `SqlConnection`
- **Python:** SQLAlchemy's `QueuePool`, `psycopg_pool`
- **Node.js:** `pg.Pool`, `mysql2` pool
- **External/proxy poolers:** PgBouncer, Pgpool-II (Postgres), ProxySQL (MySQL), RDS Proxy
