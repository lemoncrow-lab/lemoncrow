[assistant] Fixing a Node.js memory leak is a two-phase job: **find it**, then **fix it**. Most people jump to fixes and guess wrong, so lead with measurement.

## 1. Confirm it's actually a leak

Watch memory over time under sustained load. A leak means memory climbs and *never* comes back down after GC — not just that it's high.

```bash
node --expose-gc your-app.js
```

Log it periodically:
```js
setInterval(() => {
  const m = process.memoryUsage();
  console.log(`rss=${(m.rss/1e6)|0}MB heap=${(m.heapUsed/1e6)|0}MB ext=${(m.external/1e6)|0}MB`);
}, 10000);
```

- `heapUsed` growing → JS object leak (find it with heap snapshots).
- `external` / `rss` growing while heap is flat → Buffers, native addons, or C++ land.

## 2. Locate it with heap snapshots (the key step)

```bash
node --inspect your-app.js
```
Open `chrome://inspect` → Memory tab. Then:

1. Take a snapshot.
2. Drive traffic / let it run.
3. Take a second snapshot.
4. Use **"Comparison"** view — sort by delta. The object type with the largest growing count between snapshots is your leak.
5. Click it, follow the **retainers** chain to see *what's holding the reference*. That's the actual bug.

For production, `require('v8').writeHeapSnapshot()` on a signal, then analyze offline.

## 3. Fix the common causes

Almost all Node leaks are one of these — something keeps a reference alive:

| Cause | Fix |
|---|---|
| Listeners added in a loop / per-request, never removed | `emitter.removeListener()`, or check for `MaxListenersExceededWarning` |
| Growing global/module-level `Map`/array/cache | Add eviction (LRU, TTL) or use `WeakMap`/`WeakRef` for keyed-by-object data |
| `setInterval`/`setTimeout` never cleared | `clearInterval`; call `.unref()` if it shouldn't hold the process open |
| Closures capturing large objects | Null out references; narrow closure scope |
| Unbounded queues / buffering (backpressure) | Respect stream backpressure; cap queue size |
| Caching request/response objects | Never retain req/res past the request lifecycle |

## Quick reality checks

- **Global collections** are the #1 culprit — grep for module-level `new Map()`, `[]`, or object caches that only ever grow.
- **`--max-old-space-size`** raises the ceiling but does **not** fix a leak; it just delays the crash. Only bump it if memory genuinely plateaus.
- Reproduce in a load test (`autocannon`, `k6`) so you can iterate fast instead of waiting hours in prod.
