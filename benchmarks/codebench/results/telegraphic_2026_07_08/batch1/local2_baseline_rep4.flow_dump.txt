[assistant] Database connection pooling is a technique for managing and reusing database connections rather than opening a new one for every request. Here's how it works and why it matters.

## The problem it solves

Opening a database connection is expensive. Each new connection involves:

- A TCP handshake (and often a TLS handshake)
- Authentication (username/password, credential validation)
- Backend resource allocation on the database server (memory, a process or thread)

This setup can take tens of milliseconds — often far longer than the actual query. If your app opens and closes a connection per request, that overhead dominates, and under load you can exhaust the database's connection limit entirely.

## How pooling works

A connection pool is a cache of open, ready-to-use connections maintained by your application (or a middleware layer).

```
1. On startup, the pool opens N connections and keeps them idle.
2. When code needs the DB, it "borrows" a connection from the pool.
3. It runs its queries.
4. Instead of closing, it "returns" the connection to the pool.
5. The next request reuses that same warm connection.
```

The connection is never really closed during normal operation — it's just lent out and reclaimed.

## Key configuration parameters

| Parameter | Meaning |
|-----------|---------|
| **Min pool size** | Connections kept open even when idle |
| **Max pool size** | Upper limit on concurrent connections |
| **Connection timeout** | How long a caller waits for a free connection before erroring |
| **Idle timeout** | How long an unused connection stays open before being closed |
| **Max lifetime** | Force-recycle a connection after this age (avoids stale/leaked server state) |

**Sizing the max is the most important decision.** Bigger is not better — each connection consumes resources on the database server. A common starting formula is roughly `connections = ((core_count * 2) + effective_spindle_count)`, but the real answer comes from load testing. Many small apps run fine with a pool of 10–20.

## What happens under saturation

When all connections are in use, new requests **queue** (up to the connection timeout) rather than opening unlimited connections. This is a feature: it applies backpressure and protects the database from being overwhelmed, trading some request latency for overall stability.

## Practical notes

- **Use an existing library.** HikariCP (Java), `pgbouncer` (Postgres, external), SQLAlchemy's pool (Python), `database/sql`'s built-in pool (Go), node-postgres `Pool` (Node). Don't build your own.
- **Serverless/edge is a special case.** Many short-lived function instances each holding a pool can blow past DB limits. Use an external pooler like PgBouncer (in transaction mode) or a data proxy (e.g., RDS Proxy, Prisma Accelerate) that pools centrally.
- **Always return connections** — a leaked connection (borrowed but never returned) permanently shrinks the pool. Use language constructs (`try/finally`, context managers, `defer`) to guarantee return.
- **Validate on borrow** if your network drops idle connections; a `maxLifetime` shorter than the DB/firewall idle timeout avoids handing out dead connections.

## The trade-off in one line

Pooling trades a small amount of memory and complexity for a large reduction in per-request latency and a hard cap that shields your database from connection exhaustion.
