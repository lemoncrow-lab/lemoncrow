[assistant] A hash table handles collisions — when two different keys hash to the same bucket index — using one of two main families of strategies:

## 1. Separate Chaining
Each bucket holds a *secondary data structure* (usually a linked list, sometimes a dynamic array or balanced tree) containing all entries that hash to that index.

- **Insert:** compute the index, append the key/value to that bucket's list.
- **Lookup:** compute the index, then scan the list comparing keys.

```
index 3 → [ ("cat",1) ] → [ ("dog",2) ]   # both hashed to 3
```

- Simple; the table never "fills up."
- Performance degrades to O(n) if many keys land in one bucket. Java's `HashMap` mitigates this by converting a long chain into a balanced tree (O(log n)).

## 2. Open Addressing
All entries live *in the array itself*. On collision, you *probe* for another open slot using a defined sequence:

- **Linear probing:** try `i+1, i+2, i+3, …` (cache-friendly, but causes "clustering").
- **Quadratic probing:** try `i+1², i+2², …` (reduces clustering).
- **Double hashing:** step size determined by a second hash function (best distribution).

Lookup follows the same probe sequence until it finds the key or an empty slot. Deletion needs care — you use a "tombstone" marker so probe chains aren't broken.

## Key supporting factors

- **Load factor** (entries ÷ buckets): when it exceeds a threshold (commonly ~0.75), the table **resizes and rehashes** all entries into a larger array to keep operations near O(1). This is essential for open addressing, which fails when full.
- **Good hash function:** distributing keys uniformly is the first line of defense — fewer collisions means fewer to resolve.

## Quick comparison

| | Separate chaining | Open addressing |
|---|---|---|
| Storage | External lists | In-array |
| Load factor | Can exceed 1 | Must stay < 1 |
| Cache locality | Worse | Better |
| Deletion | Easy | Needs tombstones |

**In short:** collisions are inevitable (pigeonhole principle), so hash tables either *chain* colliding entries together in each bucket or *probe* for alternative slots — while keeping the load factor low via resizing to preserve average O(1) performance.
