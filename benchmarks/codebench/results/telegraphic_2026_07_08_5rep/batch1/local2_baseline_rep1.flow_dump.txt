[assistant] Database connection pooling is a technique for managing and reusing database connections so that applications don't pay the cost of opening a new connection for every query.

## The problem it solves

Opening a database connection is expensive. Each new connection typically requires:

- A TCP handshake (and often a TLS handshake)
- Authentication (verifying username/password)
- Session and memory allocation on the database server

This can take tens to hundreds of milliseconds. If your app opens and closes a connection for every request, that overhead dominates, and under load you can exhaust the database's connection limit entirely.

## How pooling works

A connection pool is a cache of open, ready-to-use connections maintained by your application (or a middleware layer).

1. **On startup**, the pool opens a set of connections and keeps them idle.
2. **When code needs a connection**, it *borrows* one from the pool instead of creating a new one.
3. **When done**, the code *returns* the connection to the pool rather than closing it.
4. The connection stays open, ready for the next borrower.

```
App threads          Pool                 Database
  [T1] ──borrow──►  [ conn1 ] ──────────►  ┌────────┐
  [T2] ──borrow──►  [ conn2 ] ──────────►  │   DB   │
  [T3] ──wait...    [ conn3 ] (idle)       └────────┘
                    [ conn4 ] (idle)
```

## Key configuration parameters

| Setting | Meaning |
|---|---|
| **Min / initial size** | Connections kept open even when idle |
| **Max size** | Upper limit; borrowers wait when all are in use |
| **Connection timeout** | How long a borrower waits before failing |
| **Idle timeout** | Close connections unused for this long |
| **Max lifetime** | Retire connections after a set age (avoids stale/leaked server-side state) |
| **Validation / test query** | Check a connection is alive before handing it out |

## Benefits

- **Lower latency** — no per-request handshake/auth cost
- **Bounded load** — the max size protects the database from being overwhelmed
- **Better throughput** — reuse means more queries per second
- **Predictable resource use** — connections are a finite, controlled resource

## Sizing and gotchas

- **Bigger is not better.** A common formula is roughly `connections ≈ ((core_count * 2) + effective_spindle_count)`. Too many connections cause context-switching and lock contention on the DB, often *reducing* throughput.
- **Connection leaks** — if code borrows but forgets to return (e.g., missing `finally`/`using`/context manager), the pool drains and everything stalls. Always release in a `finally` block or use language constructs that auto-return.
- **Serverless / many instances** — each app instance has its own pool, so `instances × max_size` can blow past the DB's limit. This is where an external pooler like **PgBouncer** (transaction-level pooling) helps.
- **Stale connections** — networks and databases drop idle connections; `max lifetime` and validation queries guard against handing out dead ones.

## Where it lives

- **In-process libraries**: HikariCP (Java), `database/sql` (Go), SQLAlchemy's `QueuePool` (Python), `pg-pool` (Node).
- **External/sidecar poolers**: PgBouncer, Pgpool-II (Postgres), ProxySQL (MySQL) — useful when many app instances share one database.

**In short:** connection pooling trades a small amount of memory (keeping connections open) for large savings in latency and stability by reusing a controlled set of connections instead of constantly creating and tearing them down.
