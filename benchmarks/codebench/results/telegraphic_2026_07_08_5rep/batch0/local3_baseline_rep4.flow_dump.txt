[assistant] Here's how to set up a robust PostgreSQL connection pool in Node.js using the standard `pg` library.

## Installation

```bash
npm install pg
```

## Pool Configuration

```javascript
// db.js
const { Pool } = require('pg');

const pool = new Pool({
  host: process.env.PGHOST || 'localhost',
  port: process.env.PGPORT || 5432,
  database: process.env.PGDATABASE,
  user: process.env.PGUSER,
  password: <redacted>,

  // --- Pool sizing ---
  max: 20,                       // max clients in the pool
  min: 2,                        // keep a few warm (pg >= 8.10)

  // --- Timeouts ---
  connectionTimeoutMillis: 5000, // fail if a connection can't be acquired in 5s
  idleTimeoutMillis: 30000,      // close idle clients after 30s
  maxUses: 7500,                 // recycle a client after N queries (avoids leaks)

  // --- Per-query / statement timeouts (server-enforced) ---
  statement_timeout: 10000,      // abort any statement running > 10s
  query_timeout: 10000,          // client-side query timeout
  idle_in_transaction_session_timeout: 10000, // kill sessions idle in a txn

  // --- Keep-alive (helps with load balancers / NAT dropping idle TCP) ---
  keepAlive: true,
  keepAliveInitialDelayMillis: 10000,

  // --- TLS (for managed DBs like RDS, Supabase, etc.) ---
  ssl: process.env.PGSSL === 'true'
    ? { rejectUnauthorized: true }  // provide `ca` for self-signed certs
    : false,
});
```

## Error Handling

The critical piece most people miss: **the pool emits `error` on idle clients**. If you don't listen, an unhandled error will crash your process.

```javascript
// Handle errors on idle clients in the pool.
// Without this listener, a backend dropping an idle connection crashes Node.
pool.on('error', (err) => {
  console.error('Unexpected error on idle PostgreSQL client', err);
  // Don't exit here — the pool recovers by creating new clients.
  // Log/alert instead.
});

pool.on('connect', (client) => {
  // Optional: run session setup per new physical connection
  client.query('SET application_name = $1', ['my-app']).catch(() => {});
});
```

## Querying Safely

For simple, one-shot queries, use `pool.query()` — it checks out a client and releases it automatically:

```javascript
async function getUser(id) {
  // Always use parameterized queries — never string interpolation (SQL injection).
  const { rows } = await pool.query('SELECT * FROM users WHERE id = $1', [id]);
  return rows[0];
}
```

## Transactions (manual checkout — release is mandatory)

When you check out a client manually, you **must** release it in a `finally`, or you leak connections until the pool is exhausted:

```javascript
async function transferFunds(fromId, toId, amount) {
  const client = await pool.connect();
  try {
    await client.query('BEGIN');
    await client.query('UPDATE accounts SET balance = balance - $1 WHERE id = $2', [amount, fromId]);
    await client.query('UPDATE accounts SET balance = balance + $1 WHERE id = $2', [amount, toId]);
    await client.query('COMMIT');
  } catch (err) {
    await client.query('ROLLBACK');
    throw err;
  } finally {
    client.release(); // always — even on error
  }
}
```

## Graceful Shutdown

Drain the pool on shutdown so in-flight queries finish and sockets close cleanly:

```javascript
async function shutdown() {
  console.log('Draining PostgreSQL pool...');
  await pool.end(); // waits for active clients, then closes all
  process.exit(0);
}

process.on('SIGTERM', shutdown);
process.on('SIGINT', shutdown);
```

## Key points

| Setting | Why it matters |
|---|---|
| `pool.on('error')` | **Non-optional** — prevents crashes when idle backends drop |
| `connectionTimeoutMillis` | Fail fast instead of hanging when the DB is unreachable |
| `statement_timeout` | Server-side guard against runaway queries holding connections |
| `idle_in_transaction_session_timeout` | Prevents an app bug from locking rows indefinitely |
| `client.release()` in `finally` | Prevents connection leaks that exhaust the pool |
| `max` sizing | Total across all app instances must stay under Postgres's `max_connections` |
| `maxUses` | Guards against slow memory/resource creep on long-lived connections |

**On `max` sizing:** if you run 4 app instances with `max: 20`, that's 80 connections. Make sure that plus overhead stays under the server's `max_connections` (default 100). For serverless/high-concurrency, put **PgBouncer** in front and keep the app-side pool small.
