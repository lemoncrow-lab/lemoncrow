[assistant] This route handler has a **critical SQL injection vulnerability**, plus a couple of secondary issues.

## 1. SQL Injection (critical)

```js
const query = `SELECT * FROM users WHERE id = ${req.params.id}`;
```

`req.params.id` is interpolated directly into the SQL string. An attacker fully controls it. For example:

- `GET /api/users/1 OR 1=1` → returns **every user** in the table.
- `GET /api/users/0 UNION SELECT username, password, ... FROM users` → exfiltrate arbitrary columns/tables.
- `GET /api/users/1; DROP TABLE users--` → destructive, if the driver allows stacked queries.

**Fix — use parameterized queries** so the value can never be parsed as SQL:

```js
app.get('/api/users/:id', (req, res) => {
  const query = 'SELECT * FROM users WHERE id = ?';   // or $1 for pg
  db.query(query, [req.params.id])
    .then(user => res.json(user))
    .catch(err => {
      console.error(err);
      res.status(500).json({ error: 'Internal server error' });
    });
});
```

Never build SQL with string interpolation/concatenation from request data. If you must inline something structural (e.g. a column name), validate it against an allowlist — parameters only bind values, not identifiers.

## 2. No error handling (info disclosure + hang)

The `.then()` has no `.catch()`. A rejected DB promise becomes an **unhandled rejection**, and the request hangs until it times out. If a global handler leaks the error, raw DB/driver messages can reach the client and aid an attacker. The `.catch()` above fixes this — log server-side, return a generic message client-side.

## 3. Secondary hardening

- **`SELECT *`** returns every column, including sensitive ones (`password_hash`, tokens, internal flags) that then get serialized straight to the client via `res.json(user)`. Select an explicit column allowlist instead.
- **Input validation**: if IDs are integers, validate/coerce before querying (e.g. reject non-numeric `id` with a `400`). Defense-in-depth beyond parameterization, and it also gives cleaner errors.
- **AuthZ**: nothing here checks *who* is asking. Confirm the caller is authenticated and authorized to view the requested user — otherwise this is an IDOR (any user can enumerate any other user by ID).

### Hardened version

```js
app.get('/api/users/:id', requireAuth, (req, res) => {
  const id = Number(req.params.id);
  if (!Number.isInteger(id) || id <= 0) {
    return res.status(400).json({ error: 'Invalid user id' });
  }

  const query = 'SELECT id, username, email, created_at FROM users WHERE id = ?';
  db.query(query, [id])
    .then(user => {
      if (!user) return res.status(404).json({ error: 'Not found' });
      res.json(user);
    })
    .catch(err => {
      console.error(err);
      res.status(500).json({ error: 'Internal server error' });
    });
});
```

The SQL injection is the one that matters most — fix that first with parameterized queries; the rest is hardening.
