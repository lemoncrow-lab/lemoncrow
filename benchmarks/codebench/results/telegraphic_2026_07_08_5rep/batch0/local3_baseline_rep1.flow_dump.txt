[assistant] Here's how to set up a robust PostgreSQL connection pool in Node.js using the `pg` library (node-postgres), which is the standard choice.

## Installation

```bash
npm install pg
```

## Basic Pool Setup with Full Configuration

```javascript
// db.js
const { Pool } = require('pg');

const pool = new Pool({
  // Connection
  host: process.env.PGHOST || 'localhost',
  port: process.env.PGPORT || 5432,
  database: process.env.PGDATABASE,
  user: process.env.PGUSER,
  password: <redacted>,

  // Pool sizing
  max: 20,                        // max clients in the pool
  min: 2,                         // (pg-pool doesn't enforce min directly; see note below)

  // Timeouts (all in milliseconds)
  connectionTimeoutMillis: 5000,  // fail if a connection can't be acquired in 5s
  idleTimeoutMillis: 30000,       // close idle clients after 30s
  maxUses: 7500,                  // recycle a connection after N uses (avoids memory leaks)

  // Query-level timeouts (applied per client)
  statement_timeout: 10000,       // abort any statement running > 10s
  query_timeout: 10000,           // client-side query timeout
  idle_in_transaction_session_timeout: 10000, // kill idle-in-transaction sessions

  // TLS (for managed/cloud Postgres like RDS, Supabase, etc.)
  ssl: process.env.PGSSL === 'true'
    ? { rejectUnauthorized: false } // set to a CA cert in production
    : false,
});

module.exports = pool;
```

## Error Handling

The single most important thing people miss: **the pool emits `error` for idle clients**. If you don't handle it, an unexpected backend/network error will crash your process.

```javascript
pool.on('error', (err, client) => {
  // Fired when an idle client experiences an error (e.g. network drop,
  // DB restart). The pool removes that client automatically.
  console.error('Unexpected error on idle PostgreSQL client', err);
  // Do NOT exit the process — the pool recovers. Just log/alert.
});

pool.on('connect', (client) => {
  // Optional: runs each time a new physical connection is established
});
```

## Safe Query Helpers

Use the pool's `query()` for simple, single statements — it checks out and returns a client automatically:

```javascript
async function query(text, params) {
  const start = Date.now();
  const res = await pool.query(text, params);
  const duration = Date.now() - start;
  if (duration > 1000) {
    console.warn('Slow query', { text, duration, rows: res.rowCount });
  }
  return res;
}
```

For **transactions or multi-statement work**, check out a client and always release it in `finally`:

```javascript
async function withTransaction(callback) {
  const client = await pool.connect();
  try {
    await client.query('BEGIN');
    const result = await callback(client);
    await client.query('COMMIT');
    return result;
  } catch (err) {
    await client.query('ROLLBACK');
    throw err;
  } finally {
    client.release(); // ALWAYS release, even on error — else you leak the connection
  }
}

// Usage
await withTransaction(async (client) => {
  await client.query('UPDATE accounts SET balance = balance - $1 WHERE id = $2', [100, 1]);
  await client.query('UPDATE accounts SET balance = balance + $1 WHERE id = $2', [100, 2]);
});
```

## Graceful Shutdown

Drain the pool on shutdown so in-flight queries finish and connections close cleanly:

```javascript
async function shutdown() {
  console.log('Closing PostgreSQL pool...');
  await pool.end(); // waits for active queries, then closes all connections
  process.exit(0);
}

process.on('SIGINT', shutdown);
process.on('SIGTERM', shutdown);
```

## Key Configuration Notes

| Setting | Why it matters |
|---|---|
| `connectionTimeoutMillis` | Without it, `pool.connect()` hangs forever if the pool is exhausted or the DB is unreachable. |
| `idleTimeoutMillis` | Frees idle connections so you don't hold DB resources unnecessarily. |
| `statement_timeout` | Server-side guard against runaway queries — the best defense. |
| `maxUses` | Recycles connections periodically; helps with load balancers and avoids rare memory growth. |
| `max` | Size to your DB's `max_connections` divided across app instances. Don't set it huge — Postgres connections are expensive. Consider **PgBouncer** if you need many. |
| `pool.on('error')` | Mandatory — prevents crashes from idle-client failures. |

## Sizing Guidance

A common mistake is setting `max` too high. If Postgres allows 100 connections and you run 5 app instances, keep `max` around 15–20 each (leaving headroom for migrations/admin). For serverless or high-concurrency workloads, put **PgBouncer** in front and let the pool be small.
