[assistant] Both `git merge` and `git rebase` integrate changes from one branch into another, but they do it in fundamentally different ways.

## The core difference

**Merge** combines two branches by creating a new "merge commit" that ties their histories together. Your branch's history is preserved exactly as it happened.

```
      A---B---C  feature
     /         \
D---E---F---G---M  main   (M is the merge commit)
```

**Rebase** replays your branch's commits one-by-one on top of another branch, creating brand-new commits with new hashes. It rewrites history to make it look linear.

```
              A'--B'--C'  feature (replayed onto G)
             /
D---E---F---G  main
```

## Mental models

- **Merge** = "record what actually happened," including the fact that two lines of work happened in parallel.
- **Rebase** = "tell the story as if I did my work starting from the latest main."

## When to use each

**Use merge when:**
- Integrating a finished feature branch into a shared branch (`main`, `develop`). A merge commit is a useful record of "this feature landed here."
- The branch is shared — other people have based work on it, or it's already pushed.
- You value an accurate, auditable history over a tidy one.

**Use rebase when:**
- Updating your *local, private* feature branch with the latest `main` before you're done, so you're working against current code (`git rebase main`).
- Cleaning up your own commits before sharing (`git rebase -i` to squash, reorder, reword).
- You want a clean, linear history that's easy to read and `git bisect`.

A very common combined workflow: **rebase your local branch to keep it current, then merge it into main** (often a `--no-ff` merge) when it's ready.

## Tradeoffs

| | Merge | Rebase |
|---|---|---|
| History | True, but can get noisy with many merge commits | Clean, linear, easy to follow |
| Commit hashes | Preserved | Rewritten (new commits) |
| Conflict handling | Resolve once, in the merge commit | May resolve conflicts repeatedly, per replayed commit |
| Safety | Non-destructive | Rewrites history — risky if shared |
| Traceability | Merge commit shows when/where work integrated | That context is lost |

## The golden rule

**Never rebase commits that others may have already pulled** (e.g. a pushed shared branch). Rebasing rewrites commits, so anyone who based work on the originals ends up with a divergent, conflicting history. Keep rebasing to your own local, unpublished work.

If you've already pushed a branch that only you use and you rebase it, you'll need `git push --force-with-lease` — which is safer than `--force` because it refuses to overwrite if someone else pushed in the meantime.

## Quick heuristics
- Solo cleanup / staying current with main → **rebase**
- Landing shared work / preserving real history → **merge**
- Branch already shared and pulled by others → **merge** (don't rewrite)
