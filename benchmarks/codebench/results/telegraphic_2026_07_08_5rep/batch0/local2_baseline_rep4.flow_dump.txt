## The bug: seconds vs. milliseconds

JWT's `exp` claim is a **NumericDate** (RFC 7519) — **seconds** since the Unix epoch (e.g. `1751979600`). But `Date.now()` returns **milliseconds** (e.g. `1751979600000`). They differ by a factor of 1000, so any raw comparison between them is meaningless — `Date.now()` is ~1000× larger than `exp` for every token.

Depending on how your comparison is written, that mismatch makes the expiry branch fire for *all* tokens or *no* tokens — and if your "expired" condition never evaluates true, expired tokens sail straight through.

## Fix (if you keep the manual check)

Convert to the same unit and get the direction right — expired means **now is at or past exp**:

```js
// Option A: scale exp up to ms
if (Date.now() >= payload.exp * 1000) {
  return res.status(401).json({ error: 'Token expired' });
}

// Option B: scale now down to seconds
if (Math.floor(Date.now() / 1000) >= payload.exp) {
  return res.status(401).json({ error: 'Token expired' });
}
```

Two adjacent traps to check while you're in there:

1. **Comparison direction** — use `>=` (now past exp = expired), not `<`.
2. **Missing `exp`** — if a token has no `exp`, `payload.exp * 1000` is `NaN`, and `Date.now() >= NaN` is `false`, so the token is treated as *never expiring*. Reject explicitly when `exp` is absent:

```js
if (typeof payload.exp !== 'number') {
  return res.status(401).json({ error: 'Token missing expiry' });
}
```

## Better: don't hand-roll it

`jsonwebtoken`'s `jwt.verify` validates `exp` for you — it handles the seconds conversion internally and throws `TokenExpiredError` on an expired token. It also verifies the signature, which a manual `exp` check skips entirely (a decode-only path will happily trust a forged/tampered token):

```js
const jwt = require('jsonwebtoken');

function authMiddleware(req, res, next) {
  const token = req.headers.authorization?.split(' ')[1];
  if (!token) return res.status(401).json({ error: 'No token' });

  try {
    req.user = jwt.verify(token, process.env.JWT_SECRET); // throws on expiry or bad signature
    next();
  } catch (err) {
    const msg = err.name === 'TokenExpiredError' ? 'Token expired' : 'Invalid token';
    return res.status(401).json({ error: msg });
  }
}
```

If you're currently using `jwt.decode()` (which does **not** verify signature or expiry) anywhere in this path, that's the deeper issue — switch it to `jwt.verify()`.
