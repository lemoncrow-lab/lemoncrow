[assistant] Here's a practical playbook for tracking down and fixing memory leaks in a long-running Node.js process.

## 1. Confirm it's actually a leak

Run with GC exposed and watch heap growth over time:

```bash
node --expose-gc --max-old-space-size=512 app.js
```

Log heap usage periodically:

```js
setInterval(() => {
  const m = process.memoryUsage();
  console.log(`rss=${(m.rss/1e6).toFixed(0)}MB heapUsed=${(m.heapUsed/1e6).toFixed(0)}MB external=${(m.external/1e6).toFixed(0)}MB`);
}, 10_000);
```

A real leak = heap that trends upward across GC cycles and never returns to baseline under steady load. Sawtooth that returns to baseline is normal.

## 2. Capture and compare heap snapshots

This is the single most effective technique. Take a snapshot, apply load, take another, and diff them.

```js
const v8 = require('v8');
const fs = require('fs');
// call this on a signal or timer
function snapshot() {
  const name = `/tmp/heap-${process.hrtime.bigint()}.heapsnapshot`;
  fs.writeFileSync(name, v8.getHeapSnapshot ? '' : '');
  v8.writeHeapSnapshot(name);
}
```

Then open both `.heapsnapshot` files in **Chrome DevTools** (chrome://inspect → Memory tab → Load). Use the **Comparison** view between snapshot 1 and 2 — sort by "Delta" to see which object types are accumulating (retained count going up = your leak source). The "Retainers" panel shows *what* is holding the objects alive.

You can also connect live: run `node --inspect app.js` and attach DevTools directly.

## 3. The usual suspects (in order of likelihood)

- **Unbounded caches / Maps / arrays** — a module-level `Map` or array you `.push()` to but never evict. Fix: use `lru-cache`, bound the size, or use `WeakMap`/`WeakRef` where keys are objects.
- **Event listener accumulation** — calling `emitter.on()` without ever `.off()`/`removeListener()`, especially per-request. Watch for the `MaxListenersExceededWarning`. Always pair add/remove, or use `{ once: true }`.
- **Timers not cleared** — `setInterval`/`setTimeout` capturing large closures and never `clearInterval`'d.
- **Closures over large objects** — a long-lived callback captures a big buffer or request context.
- **Global/module state** — anything attached to `global`, or singletons that append.
- **External/native memory** — Buffers, DB connection pools, streams not being consumed or destroyed. Check `external` and `rss`, not just `heapUsed`.

## 4. Tooling that speeds this up

- `clinic doctor` / `clinic heapprofiler` (`npm i -g clinic`) — generates flamegraphs and heap timelines automatically.
- `node --heapsnapshot-near-heap-limit=3 --max-old-space-size=...` — auto-dumps a snapshot right before OOM, great for leaks that only show in production.
- `process.memoryUsage()` fields matter: `heapUsed` (JS objects), `external` (C++/Buffers bound to JS), `arrayBuffers` (subset of external), `rss` (total). If `rss` grows but `heapUsed` is flat, look at Buffers/native addons, not JS objects.

## 5. Verify the fix

Reproduce the leak with a load test (e.g. `autocannon` against an HTTP server), watch heap stabilize at a plateau across many GC cycles rather than climbing. Don't trust a short run — memory leaks often only manifest over hours.
