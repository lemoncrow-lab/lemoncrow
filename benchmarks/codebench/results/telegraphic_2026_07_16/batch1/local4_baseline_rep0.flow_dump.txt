[assistant] Here's a practical workflow for tracking down and fixing a Node.js memory leak.

## 1. Confirm it's actually a leak

Watch memory over time. A leak means RSS/heap grows monotonically and never comes back down after GC, even when load is steady or idle.

```js
setInterval(() => {
  const m = process.memoryUsage();
  console.log(JSON.stringify({
    rss: m.rss, heapUsed: m.heapUsed, external: m.external, arrayBuffers: m.arrayBuffers
  }));
}, 10000);
```

- `heapUsed` climbing → JS objects being retained.
- `external` / `arrayBuffers` climbing → Buffers, TypedArrays, or native addons.
- `rss` climbing but heap flat → native memory, or fragmentation.

## 2. Capture heap snapshots and diff them

This is the single most useful technique.

- Start the process with `--inspect`, open `chrome://inspect` in Chrome → Memory tab.
- Take a snapshot, exercise the app for a while, take a second snapshot.
- Use the **Comparison** view to see which object types grew, and the **retainers** panel to see what's holding them alive.

Programmatically (great for prod, on a signal):

```js
const v8 = require('v8');
process.on('SIGUSR2', () => {
  v8.writeHeapSnapshot(`/tmp/heap-${Date.now()}.heapsnapshot`);
});
```

Trigger with `kill -USR2 <pid>`, then load the `.heapsnapshot` files into Chrome DevTools.

## 3. The usual culprits

| Cause | What to look for |
|---|---|
| **Unbounded caches / Maps** | Objects added, never evicted. Use an LRU (e.g. `lru-cache`) with a max size/TTL. |
| **Listeners not removed** | `emitter.on()` in a request/loop path without `off()`. Watch for the `MaxListenersExceededWarning`. |
| **Closures capturing large state** | Callbacks/timers holding references to big objects. |
| **Timers / intervals** | `setInterval` never cleared; pending timers pin their closures. |
| **Globals & module-scope arrays** | Anything pushed to a module-level array lives forever. |
| **Streams not consumed/closed** | Backpressure buildup, unclosed handles. |
| **Native addons / Buffers** | Shows as `external`, not in JS heap snapshots. |

## 4. Reproduce and bisect

- Drive load with a tool (`autocannon`, `k6`) against a single endpoint to isolate the leaking path.
- Force GC to distinguish real retention from lazy GC: run with `--expose-gc` and call `global.gc()` between snapshots.
- Comment out / bisect suspect code paths until the growth stops.

## 5. Fix, then verify

After the fix, run the same load test and confirm heap returns to a stable baseline across GC cycles. Consider adding `--max-old-space-size=<MB>` to fail fast in testing, and a memory alert in prod so regressions surface early.
