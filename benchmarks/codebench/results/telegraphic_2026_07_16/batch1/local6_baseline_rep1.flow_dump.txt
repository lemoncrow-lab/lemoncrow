[assistant] A hash table handles collisions—cases where two distinct keys hash to the same bucket index—using one of two main strategies:

## 1. Separate Chaining
Each bucket holds a secondary data structure (usually a linked list) of all entries that hash there.

- **Insert:** Compute the index, then append/prepend to that bucket's list.
- **Lookup:** Hash to the bucket, then scan the list comparing keys with `equals`.
- **Delete:** Find and unlink the node.

```
index 3 → [ ("cat",1) ] → [ ("act",9) ] → null
```

- Simple; the table never "fills up."
- Extra memory for pointers; poor cache locality.
- Modern variants (e.g., Java's `HashMap` since 8) convert a bucket's list to a **balanced tree** once it grows past a threshold, improving worst case from O(n) to O(log n).

## 2. Open Addressing
All entries live in the array itself. On collision, you *probe* for another open slot using a defined sequence:

| Scheme | Probe sequence | Trade-off |
|--------|---------------|-----------|
| **Linear probing** | `h, h+1, h+2, …` | Cache-friendly, but suffers *primary clustering* |
| **Quadratic probing** | `h, h+1², h+2², …` | Reduces clustering; can miss slots unless sized carefully |
| **Double hashing** | `h, h+g, h+2g, …` (step `g` from a second hash) | Best distribution, less clustering |

- Deletions need a **tombstone** marker so probe chains aren't broken.
- Performance degrades sharply as the table fills, so a low **load factor** (α ≈ 0.5–0.7) is required.

## Key underlying ideas
- **Load factor (α = entries / buckets)** governs collision frequency. When it exceeds a threshold, the table **resizes** (typically doubles) and **rehashes** everything.
- A **good hash function** that spreads keys uniformly is the first line of defense—collisions become rare rather than needing to be resolved often.

**Rule of thumb:** chaining is more forgiving of high load factors and easy to implement; open addressing is more memory- and cache-efficient when you keep the table sparse.
