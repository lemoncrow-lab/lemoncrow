[assistant] # Database Connection Pooling

Connection pooling is a technique for **reusing a set of pre-established database connections** instead of opening and closing a new one for every query. A pool maintains a cache of open connections that application code borrows, uses, and returns.

## The problem it solves

Opening a database connection is expensive:

- **TCP handshake** + often a TLS handshake
- **Authentication** (credential check, sometimes a round trip or two)
- **Session setup** on the server (memory allocation, backend process/thread)

This can take tens of milliseconds — often far longer than the query itself. Under load, opening a fresh connection per request wastes time and can exhaust the database's connection limit (each connection consumes memory and, in Postgres, a whole backend process).

## How it works

```
                 ┌─────────────────────────┐
  app request ──▶│      Connection Pool     │
                 │  [idle] [idle] [in-use]  │──▶ Database
  app returns ◀──│  [in-use] [idle] ...     │
                 └─────────────────────────┘
```

1. On startup (or lazily), the pool opens **N** connections.
2. When code needs the DB, it **borrows** ("acquires") an idle connection.
3. It runs its queries.
4. It **returns** ("releases") the connection to the pool — the connection stays open.
5. If all connections are busy, new requests **wait** (up to a timeout) or the pool opens more, depending on config.

## Key configuration parameters

| Parameter | Meaning |
|---|---|
| **min / initial size** | Connections kept open even when idle |
| **max size** | Hard cap on concurrent connections (protects the DB) |
| **acquire timeout** | How long a caller waits for a free connection before erroring |
| **idle timeout** | Close connections unused for this long, shrinking back toward min |
| **max lifetime** | Retire a connection after this age (avoids stale connections, aids failover/load rebalancing) |
| **validation / test query** | Check a connection is still alive before handing it out |

## Sizing the pool

Bigger is **not** better. Each connection is load on the database. A common starting point:

```
pool_size ≈ ((core_count × 2) + effective_spindle_count)
```

The main constraint is the **database's** max connections, shared across *all* app instances. If you run 10 app servers each with a pool of 20, that's 200 connections hitting one database — plan against the DB's limit, not per-app.

## Common pitfalls

- **Connection leaks** — borrowing without returning (e.g., forgetting to close in an error path) slowly drains the pool until everything blocks. Always release in a `finally` block or use language constructs that auto-return.
- **Pool too large** — overwhelms the DB with context-switching and memory pressure.
- **Pool too small** — requests queue up waiting for a connection under load.
- **Long-held connections** — holding a connection across slow external calls ties up a scarce resource.
- **Serverless / many instances** — thousands of ephemeral instances each with a pool can blow past DB limits; use a **proxy pooler** (see below).

## Where it lives

- **In-app libraries**: HikariCP (Java), `database/sql` (Go), SQLAlchemy `QueuePool` (Python), `pg-pool` (Node).
- **External poolers**: PgBouncer, Pgpool-II, RDS Proxy — a middleware layer that pools on behalf of many clients, essential for serverless and high-instance-count architectures.

## Why it matters

Pooling turns a per-request cost of "open + auth + setup + query + teardown" into just "query," dramatically improving **latency** and **throughput** while **protecting the database** from connection exhaustion.
