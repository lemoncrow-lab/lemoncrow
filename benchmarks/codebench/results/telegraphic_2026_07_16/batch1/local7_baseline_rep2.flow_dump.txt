## What's actually happening

CORS (Cross-Origin Resource Sharing) errors appear when your frontend JavaScript running on one **origin** (scheme + host + port, e.g. `http://localhost:3000`) makes a request to a **different origin** (e.g. `http://localhost:8000` or `https://api.example.com`), and the server's response doesn't include the headers that authorize that cross-origin access.

Key point: **the browser blocks the response, not the request.** The server usually receives and processes the request fine — the browser just refuses to hand the response to your JS because the `Access-Control-Allow-Origin` header is missing or doesn't match. This is a server-side configuration issue, not something you fix in frontend code.

Common triggers:
- Frontend and API on different ports/hosts/schemes (very common in local dev)
- `http` vs `https` mismatch
- A preflight `OPTIONS` request (sent automatically for non-simple requests — custom headers, `PUT`/`DELETE`, JSON content-type) that the server doesn't handle
- Sending credentials/cookies without `Access-Control-Allow-Credentials: true` (and you can't use `*` for the origin in that case)

## How to fix it

**1. Fix it on the server (the correct fix).** Configure your API to send the CORS headers. Examples:

**Express (Node):**
```js
const cors = require('cors');
app.use(cors({
  origin: 'http://localhost:3000',   // your frontend origin; array or function for multiple
  credentials: true,                 // only if you send cookies/auth
}));
```

**FastAPI (Python):**
```python
from fastapi.middleware.cors import CORSMiddleware
app.add_middleware(
    CORSMiddleware,
    allow_origins=["http://localhost:3000"],
    allow_credentials=True,
    allow_methods=["*"],
    allow_headers=["*"],
)
```

**Flask:** `from flask_cors import CORS; CORS(app, origins=["http://localhost:3000"])`

**2. If you don't control the API** (third-party, or you can't change it), route through a proxy so the browser only ever talks to your own origin:
- Dev: Vite `server.proxy`, Next.js `rewrites`, or CRA `"proxy"` in package.json
- Prod: your own backend forwards the request (server-to-server has no CORS)

## Read the error carefully
The console message tells you which case you're in:
- **"No 'Access-Control-Allow-Origin' header is present"** → server isn't sending CORS headers at all
- **"...does not match... (got `*`)... credentials mode is 'include'"** → you can't use `*` with credentials; echo the specific origin instead
- **"Method ... is not allowed by Access-Control-Allow-Methods"** → add the method (and handle the `OPTIONS` preflight)
- **"Request header ... is not allowed"** → add it to `Access-Control-Allow-Headers`

## Things to avoid
- **Don't** try to fix CORS with frontend code (adding headers to your `fetch` request does nothing — the *response* headers are what matter).
- **Don't** disable web security in your browser as a "fix" — it only hides the problem on your machine.
- Browser extensions that "disable CORS" mask the real issue and won't work for your users.
