[assistant] Database connection pooling is a technique for managing and reusing database connections rather than opening and closing a new one for every request.

## The Problem It Solves

Opening a database connection is expensive. Each new connection requires:
- A TCP handshake (and often a TLS handshake)
- Authentication (username/password, sometimes cert validation)
- Session/memory allocation on the database server

This can take tens to hundreds of milliseconds — often far longer than the query itself. Under load, constantly creating and tearing down connections wastes resources and can exhaust the database's connection limit.

## How Pooling Works

A **connection pool** maintains a set of pre-established, open connections. The flow:

1. On startup, the pool opens a number of connections and keeps them idle.
2. When your app needs to query, it **borrows** a connection from the pool.
3. After the query completes, the connection is **returned** to the pool — not closed.
4. The next request reuses that same physical connection.

```
App request ─▶ [ Pool ] ─▶ borrow idle conn ─▶ run query ─▶ return conn
                  │
                  └─ maintains N warm connections
```

If all connections are in use, new requests either **wait** in a queue until one frees up, or the pool opens additional connections (up to a max), or they time out.

## Key Configuration Parameters

| Setting | Meaning |
|---|---|
| **Min pool size** | Connections kept open even when idle |
| **Max pool size** | Upper limit — caps load on the DB |
| **Connection timeout** | How long a request waits for a free connection |
| **Idle timeout** | When to close surplus idle connections |
| **Max lifetime** | Retire connections after a set age (avoids stale/leaked state) |
| **Validation query** | Health-check before handing out a connection |

## Why It Matters

- **Performance**: eliminates per-request connection setup cost.
- **Throughput**: reuse means more queries served with fewer resources.
- **Stability**: the max size protects the database from being overwhelmed by connection storms.
- **Predictability**: bounded, controlled resource usage.

## Common Implementations

- **Java**: HikariCP (the modern default), Apache DBCP, Tomcat JDBC
- **Python**: SQLAlchemy's `QueuePool`, `psycopg2.pool`
- **Node.js**: `pg-pool`, `mysql2` pools
- **Go**: built into `database/sql` (`SetMaxOpenConns`, `SetMaxIdleConns`)
- **External/proxy poolers**: PgBouncer, ProxySQL — sit between app and DB, useful when many app instances share one database.

## Practical Gotchas

- **Sizing**: bigger isn't better. A pool larger than the DB can handle causes contention. A common starting formula for Postgres is roughly `connections = (core_count * 2) + effective_spindle_count`.
- **Leaks**: forgetting to return a connection (e.g., not closing in a `finally`/`with` block) drains the pool and causes hangs.
- **Serverless**: short-lived functions plus per-instance pools can explode total connections against the DB — this is exactly where a proxy pooler like PgBouncer or a serverless driver helps.
- **Transaction state**: a returned connection should be clean (no open transaction, reset session variables) before reuse.
