[assistant] Use a **queue** for work distribution and a **topic** for event broadcasting. The core difference is how many consumers process each message.

## Queue (point-to-point)

Each message is delivered to **exactly one** consumer. Multiple consumers can read from the same queue, but they compete — a given message goes to only one of them.

**Use when:**
- You're distributing *work* — tasks, jobs, commands
- Each message should be processed once and only once
- You want to scale throughput by adding more workers (competing consumers)
- Load leveling — the queue buffers spikes so workers drain at their own pace

**Examples:** order processing, image resizing jobs, sending emails, background task offloading.

## Topic (publish/subscribe)

Each message is delivered to **every** subscriber. Publishers don't know or care who's listening.

**Use when:**
- You're broadcasting *events* — "something happened"
- Multiple independent consumers each need their own copy
- You want to decouple producers from a growing set of consumers
- Different subsystems react to the same event differently

**Examples:** an `OrderPlaced` event that triggers billing, inventory, *and* notifications — each a separate concern needing the same event.

## Quick decision guide

| Question | Queue | Topic |
|---|---|---|
| How many consumers per message? | One | Many |
| Semantics | Command ("do this") | Event ("this happened") |
| Add consumers to... | Scale throughput | Add new reactions |
| Coupling | Sender knows there's a job to do | Publisher doesn't know subscribers |

## Practical notes

- **Naming maps to intent:** commands (imperative) → queue; events (past tense) → topic.
- **You often combine them.** A common pattern is topic → per-subscriber queue (e.g., SNS→SQS, or Kafka consumer groups, or a JMS durable subscription). The topic fans out; each subscriber's queue gives it durable, independently-scalable, competing-consumer processing. This gives you broadcast *and* work distribution.
- **Consumer groups blur the line.** Kafka has no separate "queue" primitive — a topic with one consumer group acts like a queue (partitions split among group members); multiple groups make it act like a topic. Same log, different consumer topology.
- **Durability differs by system.** Classic topics only deliver to *currently connected* subscribers unless you use durable subscriptions. Log-based systems (Kafka, Pulsar) retain messages so late/replaying consumers still get them.

**Rule of thumb:** "One thing must handle this" → queue. "Anything interested may react" → topic.
