[assistant] Database connection pooling is a technique for managing and reusing database connections so your application doesn't pay the cost of opening a new connection for every query.

## The problem it solves

Opening a database connection is expensive. Each one involves:
- A TCP handshake (and often a TLS handshake)
- Authentication (username/password, sometimes a round trip or two)
- Session/backend setup on the database server (memory allocation, process/thread spawn)

This can take tens of milliseconds — often longer than the actual query. Under load, opening and closing a connection per request wastes time and can exhaust the database's connection limit.

## How pooling works

A **pool** is a cache of open, ready-to-use connections held by your application:

1. On startup (or on demand), the pool opens N connections and keeps them alive.
2. When code needs the database, it **borrows** a connection from the pool instead of opening one.
3. When done, the connection is **returned** to the pool — not closed — so it can be reused.
4. If all connections are busy, new requests **wait** (up to a timeout) or the pool opens more, up to a max limit.

```
App threads  →  [ Pool: conn1 conn2 conn3 ... ]  →  Database
             borrow / return
```

## Key configuration knobs

| Setting | What it controls |
|---|---|
| **min / initial size** | Connections kept open even when idle |
| **max size** | Hard cap on concurrent connections (protects the DB) |
| **acquire/wait timeout** | How long a caller waits for a free connection before erroring |
| **idle timeout** | Close connections unused for this long, to free resources |
| **max lifetime** | Recycle connections after a max age (avoids stale/leaked state) |
| **validation / test query** | Check a connection is still alive before handing it out |

## Why it matters

- **Latency**: reuse avoids repeated handshake/auth cost.
- **Throughput**: more requests served with fewer resources.
- **Stability**: the max size caps load on the database — a common cause of outages is thousands of app instances each opening unbounded connections.

## Sizing caveat

Bigger pools are **not** always better. A database has finite CPUs and disks; too many concurrent connections cause contention and context-switching that *reduce* throughput. A common starting heuristic is `connections ≈ (core_count × 2) + effective_spindle_count`, then tune with load testing. Also account for total connections across *all* app instances against the DB's server-side limit (e.g. Postgres `max_connections`).

## In practice

You rarely build this yourself — you use a library or a standalone proxy:
- **Language/framework pools**: HikariCP (Java), `database/sql` (Go), SQLAlchemy pool (Python), `pg`/`node-postgres` pool (Node).
- **External poolers**: PgBouncer, Pgpool-II (Postgres), ProxySQL (MySQL) — useful when many app instances or serverless functions would otherwise overwhelm the DB.

A note on **serverless/short-lived environments** (e.g. Lambda): traditional in-process pools work poorly because each instance holds its own idle connections and they scale horizontally without bound — an external pooler or a serverless-aware driver is usually the answer.
