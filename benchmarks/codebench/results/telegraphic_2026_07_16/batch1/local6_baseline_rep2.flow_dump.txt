[assistant] A hash table stores key–value pairs by mapping each key through a **hash function** to an index in an underlying array (a "bucket"). A **collision** happens when two distinct keys hash to the same index. Since collisions are unavoidable (the key space is far larger than the array), every hash table needs a strategy to resolve them. The two main families are:

## 1. Separate Chaining

Each bucket holds a secondary data structure (usually a linked list) of all entries that hashed there.

- **Insert:** compute index, append/prepend to that bucket's list.
- **Lookup:** compute index, then scan the list comparing keys (using `equals`, not just the hash).
- **Delete:** find and unlink the node.

```
index 0 → (key1, val1)
index 1 → (key2, val2) → (key5, val5)   ← collision, chained
index 2 → (key3, val3)
```

**Pros:** simple, degrades gracefully, load factor can exceed 1.
**Cons:** extra memory for pointers; poor cache locality.
Some implementations upgrade a long chain into a balanced tree — e.g. Java's `HashMap` converts a bucket to a red–black tree once it exceeds 8 entries, bounding worst-case lookup at O(log n) instead of O(n).

## 2. Open Addressing

All entries live directly in the array. On a collision, you **probe** for the next free slot according to a fixed sequence. Variants differ in how the probe sequence is computed:

| Scheme | Next slot | Issue |
|--------|-----------|-------|
| **Linear probing** | `h + 1, h + 2, …` | Primary clustering (long runs form) |
| **Quadratic probing** | `h + 1², h + 2², …` | Reduces clustering; may not visit every slot |
| **Double hashing** | `h + i·h₂(key)` | Best distribution; needs a 2nd hash function |

- **Lookup:** probe the sequence until you find the key or hit an empty slot.
- **Delete:** you can't just empty a slot — that would break probe chains — so you use a **tombstone** marker (or backward-shift the cluster).

**Pros:** no pointers, excellent cache locality.
**Cons:** performance collapses as the table fills; requires load factor well below 1 (typically resize at ~0.7).

## Keeping collisions rare

Both approaches rely on:

- **A good hash function** that distributes keys uniformly.
- **Load factor management** — when entries/buckets exceeds a threshold, the table **resizes** (usually doubles) and **rehashes** all entries, keeping average operations at **O(1)**.

The worst case is O(n) (all keys collide), but with a decent hash function and resizing, real-world hash tables give amortized O(1) insert, lookup, and delete.
