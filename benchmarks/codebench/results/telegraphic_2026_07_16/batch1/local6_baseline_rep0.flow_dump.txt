## Hash Table Collision Handling

A **collision** happens when two different keys hash to the same array index. Since you can't store two entries in one slot naively, hash tables use one of two main strategies:

### 1. Separate Chaining
Each bucket holds a secondary data structure (usually a linked list, sometimes a dynamic array or balanced tree) containing all entries that hash to that index.

```
index 3 → [ ("cat", 5) ] → [ ("dog", 9) ] → null
```

- **Insert:** hash the key, append/prepend to that bucket's list.
- **Lookup:** hash the key, then scan the list comparing full keys.
- **Pros:** simple, degrades gracefully, load factor can exceed 1.
- **Cons:** extra memory for pointers; poor cache locality.
- Java's `HashMap` uses this, and converts a bucket to a red-black tree once it grows past 8 entries to bound worst-case lookups at O(log n).

### 2. Open Addressing
All entries live in the array itself. On collision, you **probe** for another open slot using a defined sequence:

| Scheme | Next slot formula | Issue |
|--------|-------------------|-------|
| **Linear probing** | `(h + i) mod m` | Primary clustering |
| **Quadratic probing** | `(h + i²) mod m` | Secondary clustering |
| **Double hashing** | `(h₁ + i·h₂(key)) mod m` | Best distribution |

- **Lookup:** probe the same sequence until you find the key or an empty slot.
- **Deletion** is tricky — you can't just empty a slot (it would break probe chains), so you use a "tombstone" marker.
- **Pros:** great cache locality, no pointer overhead.
- **Cons:** performance degrades sharply as load factor approaches 1; typically kept below ~0.7. Python's `dict` and Ruby/Rust hash maps use open addressing.

### Keeping collisions rare
Both approaches rely on:
- **A good hash function** that distributes keys uniformly.
- **Resizing (rehashing):** when the *load factor* (entries ÷ buckets) crosses a threshold, allocate a bigger array and re-insert everything. This keeps the average chain/probe length — and thus O(1) average operations — bounded.

**In short:** collisions are inevitable (see the birthday paradox), so hash tables either store colliding keys together in a bucket (chaining) or find them a different slot in the array (open addressing), while resizing keeps collisions infrequent.
