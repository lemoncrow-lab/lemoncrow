[assistant] # Database Connection Pooling

## The Problem

Opening a database connection is expensive. Each new connection typically requires:

- A TCP handshake (and often a TLS handshake)
- Authentication (username/password verification, sometimes involving hashing rounds)
- Session/resource allocation on the database server

This can take anywhere from a few milliseconds to tens of milliseconds — an eternity if you're doing it on every single request. Databases also cap the number of concurrent connections they can handle, so opening one per request doesn't scale.

## The Solution

A **connection pool** maintains a set of pre-established, reusable database connections. Instead of opening and closing a connection for each operation, your application:

1. **Borrows** a connection from the pool
2. **Uses** it to run queries
3. **Returns** it to the pool (rather than closing it)

The connection stays alive and gets handed to the next caller who needs it.

```
┌─────────────┐        ┌──────────────────┐        ┌──────────┐
│ App Thread  │─borrow→│                  │        │          │
│ App Thread  │─borrow→│  Connection Pool │◄──────►│ Database │
│ App Thread  │─wait···│  [c1][c2][c3]    │        │          │
└─────────────┘        └──────────────────┘        └──────────┘
```

## Key Configuration Knobs

| Setting | What it controls |
|---|---|
| **Min / idle size** | Connections kept warm even when idle |
| **Max size** | Ceiling on concurrent connections (protects the DB) |
| **Acquire/wait timeout** | How long a caller waits for a free connection before failing |
| **Max lifetime** | Retire connections after some age (avoids stale/leaked server-side state) |
| **Idle timeout** | Close connections unused for a while to free resources |
| **Validation/health check** | Test a connection before handing it out (e.g. a lightweight ping) |

## Sizing the Pool

Bigger is **not** better. A common starting formula for CPU-bound workloads is roughly:

```
connections ≈ (core_count × 2) + effective_spindle_count
```

Too many connections cause context-switching and lock contention on the database, often *reducing* throughput. It's usually better to have a modest pool with a short queue than a huge pool hammering the DB.

## Common Pitfalls

- **Connection leaks** — borrowing without returning (forgetting to close/release) drains the pool until everything hangs. Use try-with-resources / context managers / `defer`.
- **Pool too small** — requests queue up and time out under load.
- **Pool too large** — overwhelms the database or exhausts its connection limit.
- **Long-held connections** — holding a connection across slow external calls starves other callers. Borrow late, return early.
- **Stale connections** — the DB or a firewall silently drops idle connections; validation and max-lifetime settings guard against this.

## Common Implementations

- **Java:** HikariCP (fast, widely used), Apache DBCP, Tomcat JDBC
- **Python:** SQLAlchemy's `QueuePool`, `psycopg_pool`
- **Node.js:** `pg.Pool`, `mysql2` pool
- **Go:** built into `database/sql` (`SetMaxOpenConns`, `SetMaxIdleConns`)
- **External/proxy:** PgBouncer, Pgpool-II (sit between app and Postgres, pooling across many app instances)

## A Note on Serverless

In serverless/autoscaling environments, each function instance may open its own pool, so total connections can explode past the DB limit. This is why external poolers (PgBouncer) or serverless-native data proxies (e.g. AWS RDS Proxy) are commonly used there.
