[assistant] Common causes and how to hunt them down:

## Find the leak first (don't guess)

```bash
node --inspect your-app.js
```

Then open `chrome://inspect`, take **heap snapshots** at intervals under load, and compare. The "Comparison" view showing which object types grew between snapshots is the fastest path to the culprit.

Alternatively, capture snapshots programmatically:

```js
const v8 = require('v8');
const fs = require('fs');
process.on('SIGUSR2', () => {
  const s = v8.getHeapSnapshot();
  s.pipe(fs.createWriteStream(`/tmp/${Date.now()}.heapsnapshot`));
});
```

Send `kill -SIGUSR2 <pid>` at intervals, load the files into Chrome DevTools.

## The usual suspects

1. **Unbounded caches / Maps** — objects added, never evicted. Use a bounded cache (e.g. `lru-cache`) or a `WeakMap` when keys are objects you don't own.

2. **Event listener accumulation** — `emitter.on(...)` called repeatedly without `removeListener`. Watch for the warning:
   ```
   MaxListenersExceededWarning: Possible EventEmitter memory leak detected
   ```
   Pair every `on` with `off`/`removeListener` on cleanup.

3. **Closures capturing large objects** — a timer/callback holds a reference to something big. Clear timers (`clearInterval`) and null out references when done.

4. **Global/module-scope arrays** that only grow — logs, request buffers, pending-job lists.

5. **Timers and pending promises** never resolved/cleared.

## Confirm and monitor

Track heap over time:

```js
setInterval(() => {
  const m = process.memoryUsage();
  console.log(`heapUsed=${(m.heapUsed/1e6).toFixed(1)}MB rss=${(m.rss/1e6).toFixed(1)}MB`);
}, 60_000);
```

Steadily climbing `heapUsed` that never drops after GC = real leak. Sawtooth that returns to baseline = normal.

Run with a lower heap cap to force the issue to surface faster in testing:
```bash
node --max-old-space-size=256 your-app.js
```
