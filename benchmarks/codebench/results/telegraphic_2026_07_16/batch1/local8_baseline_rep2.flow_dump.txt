[assistant] A debouncer on a search input solves a mismatch between how fast a user types and how expensive it is to react to each keystroke. Without it, every keystroke fires the search handler; with it, you wait until typing pauses (say 300ms) and only then act.

The concrete wins:

**1. Fewer network requests.** If a search box hits an API on every keystroke, typing "laptop" fires 6 requests. Debounced, it fires 1 — after you stop typing. That cuts server load, saves bandwidth/quota, and on metered or rate-limited APIs it's the difference between working and getting throttled.

**2. Avoids wasted work and race conditions.** Those intermediate requests ("l", "la", "lap"...) return results you never wanted to see. Worse, they can arrive out of order — the response for "lap" might land *after* the response for "laptop" and overwrite the correct results with stale ones. Firing one request sidesteps a lot of that.

**3. Smoother UI / less jank.** Even for purely local filtering (no network), re-running a filter and re-rendering a big list on every keystroke can make typing feel laggy. Debouncing lets the input stay responsive and only does the heavy work once.

**4. Cost.** If each query costs money (paid search API, LLM call, DB read units), collapsing 6 calls into 1 is a direct 6× saving.

A couple of nuances worth knowing:

- **Debounce vs. throttle.** Debounce waits for a *pause* — good for "search when they stop typing." Throttle fires at a steady max rate (e.g. once every 200ms regardless) — better for continuous events like scroll or drag. Search almost always wants debounce.

- **Pick the delay deliberately.** ~250–400ms is the usual sweet spot: long enough to batch a burst of typing, short enough to still feel instant. Too long and the results feel sluggish.

- **Cancel in-flight requests too.** Debouncing reduces how many requests you start, but pairing it with request cancellation (e.g. `AbortController`) fully closes the stale-results race.
