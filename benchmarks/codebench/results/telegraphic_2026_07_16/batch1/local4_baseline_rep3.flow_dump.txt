[assistant] Here's a practical playbook for tracking down and fixing memory leaks in a long-running Node.js process.

## 1. Confirm it's actually a leak

Watch `process.memoryUsage()` over time. A real leak shows **`heapUsed` (or `external`/`rss`) trending upward and never coming back down** after GC, across many request cycles — not just sawtoothing.

```js
setInterval(() => {
  const m = process.memoryUsage();
  console.log(JSON.stringify({
    rss: (m.rss / 1e6).toFixed(1),
    heapUsed: (m.heapUsed / 1e6).toFixed(1),
    external: (m.external / 1e6).toFixed(1),
  }));
}, 10_000);
```

Run under a steady load (e.g. `autocannon`) so growth is attributable.

## 2. Capture heap snapshots and diff them

This is the single most effective technique.

```bash
node --inspect your-app.js
```

Then open `chrome://inspect` → Memory tab. Take a snapshot, apply load, take a second snapshot, and use **"Comparison" view** to see which object types grew. Or capture programmatically:

```js
const v8 = require('v8');
// call at two points in time, then diff in DevTools
v8.writeHeapSnapshot(`./heap-${Date.now()}.heapsnapshot`);
```

In the comparison, sort by **Delta** and look at the **Retainers** tree to find what's holding references. `--inspect` plus `--expose-gc` lets you force GC between snapshots so you're only looking at truly-retained objects.

## 3. The usual culprits

| Cause | Fix |
|---|---|
| **Unbounded arrays/maps/caches** growing forever | Bound them — use `lru-cache`, TTLs, or size limits |
| **Event listeners** added per request, never removed | `removeListener`/`off`, or `AbortSignal`; watch for the `MaxListenersExceededWarning` |
| **Closures capturing large objects** held by long-lived callbacks/timers | Null out references; narrow what the closure captures |
| **Timers** (`setInterval`) never cleared | `clearInterval`; use `.unref()` where appropriate |
| **Global/module-scope state** accumulating | Prefer request-scoped state |
| **Streams** not closed/consumed (backpressure) | Handle `error`/`close`, pipe with cleanup |
| **`external` memory** growing (Buffers, native addons) | Look at Buffer pooling, native lib usage |

Global `Map`/`Set` and module-level caches are the most common — anything keyed by user/request ID that never evicts.

## 4. Reproduce in isolation

Loop the suspect operation thousands of times with `--expose-gc`, forcing GC between iterations. If memory climbs monotonically, you've localized it:

```bash
node --expose-gc repro.js
```

## 5. Mitigations while you hunt

- Set a heap ceiling so the process fails fast/restarts predictably: `node --max-old-space-size=512`.
- Use a process manager (PM2, or Kubernetes limits + liveness probe) to restart on memory thresholds — a band-aid, not a fix.
- `--heapsnapshot-near-heap-limit=2` auto-dumps a snapshot right before OOM in production.

## Fastest path

Reproduce under load → two heap snapshots → **comparison view sorted by delta** → follow the retainers to the code holding the reference → bound/evict/unsubscribe.
