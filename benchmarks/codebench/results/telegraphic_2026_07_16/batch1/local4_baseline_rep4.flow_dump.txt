[assistant] Here's a practical playbook for tracking down and fixing memory leaks in a long-running Node.js process.

## 1. Confirm it's actually a leak

Watch memory over time. A leak means memory grows monotonically and never comes back down after GC, not just occasional spikes.

```js
setInterval(() => {
  const m = process.memoryUsage();
  console.log(new Date().toISOString(),
    'rss', (m.rss / 1e6).toFixed(1),
    'heapUsed', (m.heapUsed / 1e6).toFixed(1),
    'external', (m.external / 1e6).toFixed(1));
}, 10000);
```

- **`heapUsed` grows** → JS objects being retained (the common case).
- **`external` / `rss` grows but heap is flat** → Buffers, native addons, or C++ allocations (e.g. leaked sockets, unclosed streams).

## 2. Capture heap snapshots and diff them

This is the single most effective technique.

```bash
node --inspect your-app.js
```

Open `chrome://inspect` → **Memory** tab. Take a snapshot, exercise the app / let it run, force GC, take another snapshot, then use **Comparison** view to see which object types grew. The "retainers" panel shows *what is holding the reference* — that's your leak's root.

For a headless process, capture snapshots programmatically:

```js
const v8 = require('v8');
const fs = require('fs');
// on a signal or interval
const file = v8.writeHeapSnapshot();
console.log('wrote', file); // load the .heapsnapshot into Chrome DevTools
```

## 3. Common causes to check

| Cause | Fix |
|---|---|
| Unbounded caches / `Map` / arrays that only ever grow | Add eviction (LRU, TTL) or a size cap; consider `WeakMap`/`WeakRef` for keyed-by-object caches |
| Event listeners added but never removed | `removeListener`/`off`; watch for the `MaxListenersExceededWarning` |
| `setInterval`/timers never cleared | `clearInterval`, or `.unref()` if it shouldn't hold the process open |
| Closures capturing large objects | Null out references; narrow what the closure closes over |
| Globals / module-level state accumulating | Scope data per-request instead of module scope |
| Promises that never settle | They retain their entire chain — ensure they resolve/reject |
| Leaked DB/HTTP connections or streams | Ensure `.end()`/`.destroy()`/`.close()` on all paths, including errors |

## 4. Reproduce and bisect

Drive load with something like `autocannon` or a loop, and force GC between measurements to distinguish "live" memory from garbage:

```bash
node --expose-gc your-app.js
```
```js
global.gc();
console.log(process.memoryUsage().heapUsed);
```

If memory stays high *after* `global.gc()`, it's genuinely retained.

## 5. Guardrails

- Set a heap ceiling so you get a crash + snapshot rather than an OOM kill: `node --max-old-space-size=512`.
- Consider `--heapsnapshot-near-heap-limit=2` to auto-dump a snapshot right before OOM.
- In production, watch for the `warning` event and `MaxListenersExceededWarning`.
