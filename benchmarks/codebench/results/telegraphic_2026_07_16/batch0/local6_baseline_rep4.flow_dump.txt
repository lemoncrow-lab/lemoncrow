[assistant] This is one of those decisions where the framing matters more than the answer. Microservices are a solution to *organizational and scaling* problems, not performance problems per se. Before you split anything, work through these.

## First: is "slow" even a monolith problem?

Most Django slowness is fixable in place, and splitting won't touch the real cause. Rule these out first — they're cheaper than a rewrite by an order of magnitude:

- **N+1 queries** — the #1 Django killer. Add `django-debug-toolbar` or `nplusone`, look for missing `select_related`/`prefetch_related`.
- **Missing DB indexes** — check `pg_stat_statements` for your slow queries.
- **No caching layer** — Redis/memcached for expensive reads, per-view or fragment caching.
- **Synchronous work that should be async** — offload to Celery (emails, PDF generation, third-party API calls).
- **Under-provisioned DB or app servers** — sometimes it's just capacity.

If you haven't profiled, do that before the architecture debate. Microservices will make an unprofiled slow system into a slow *distributed* system, which is strictly worse to debug.

## If it's genuinely an architecture ceiling, weigh these factors

**1. Organizational readiness (the real driver)**
Microservices are primarily about letting teams deploy independently. If you have one team of 5, you'll get all the distributed-systems cost and none of the org benefit. Conway's Law cuts both ways — service boundaries that don't match team boundaries create constant cross-team coupling.

**2. Domain boundaries**
Can you actually draw clean seams? Look for bounded contexts where data ownership is clear. If two "services" would need to share the same tables or participate in the same transactions constantly, they aren't separate services — they're one service with a network call in the middle.

**3. Data and transactions**
This is the hardest part. A monolith gives you ACID transactions for free. Split the DB and you lose them — now you need sagas, eventual consistency, idempotency, and distributed-failure handling. Joins across services become application-code orchestration.

**4. Operational maturity**
Do you have: centralized logging, distributed tracing, per-service CI/CD, service discovery, on-call rotation, containerization/orchestration? If these don't exist, you'll build a lot of plumbing before delivering any feature value.

**5. Latency and failure modes**
In-process calls become network calls: slower, and they can fail partially. You need timeouts, retries, circuit breakers. Sometimes the distributed version is *slower* under load than the monolith was.

## The pragmatic path most teams should take

1. **Profile and fix the monolith first.** Usually resolves the "slow" complaint.
2. **Modularize in place** — clean internal boundaries (Django apps with explicit interfaces, no cross-app model reach-in). This gives you most of the maintainability benefit with none of the network cost.
3. **Scale the monolith horizontally** — it's stateless if you did it right; run more instances behind a load balancer. Split reads to replicas.
4. **Extract one service** only if a specific piece has a *genuinely different* scaling profile, tech need, or team owner (e.g. a CPU-heavy ML component, or a high-write ingestion pipeline). Strangler-fig pattern: carve one piece, learn, then decide about the next.

The strongest signals you're *actually* ready to split: multiple teams stepping on each other at deploy time, one component with a wildly different resource profile than the rest, and you already have the observability/CI infrastructure to run distributed services.
