[assistant] Standard fix: make the DB do the increment atomically instead of read-then-write in JS.

**The bug** (read-modify-write race):
```js
const { rows } = await client.query('SELECT count FROM counters WHERE id=$1', [id]);
const next = rows[0].count + 1;
await client.query('UPDATE counters SET count=$1 WHERE id=$2', [next, id]);
```
Two requests read the same value → both write the same `next`.

**Fix** — single atomic statement, return the new value:
```js
const { rows } = await client.query(
  'UPDATE counters SET count = count + 1 WHERE id=$1 RETURNING count',
  [id]
);
const value = rows[0].count;
```
`count = count + 1` takes a row lock; concurrent updates serialize → each gets a distinct value.

If read + increment span multiple statements you actually need together, wrap in a txn with `SELECT ... FOR UPDATE`. But for a plain counter, the one-liner above is enough.

Share the handler if you want it patched.
