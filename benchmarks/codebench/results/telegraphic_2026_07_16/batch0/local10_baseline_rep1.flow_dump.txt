[assistant] Created `ErrorBoundary.tsx`. It's a class component (required — error boundaries can't be functions since they rely on `getDerivedStateFromError`/`componentDidCatch`).

**What it does:**
- **Catches render errors** — `getDerivedStateFromError` stores the error and triggers the fallback render; catches errors in rendering, lifecycles, and constructors of the subtree.
- **Fallback UI with retry** — default `DefaultFallback` shows the message and a "Try again" button that calls `retry()`, which clears the error state and re-renders `children`.
- **Logs details** — `componentDidCatch` logs the error and component stack to the console, and forwards to an optional `onError` callback for wiring up a reporting service.

**Usage:**

```tsx
// Default fallback
<ErrorBoundary onError={(err, info) => reportToSentry(err, info)}>
  <MyApp />
</ErrorBoundary>

// Custom fallback via render prop (gets error + retry)
<ErrorBoundary fallback={({ error, retry }) => (
  <div>
    <p>Failed: {error.message}</p>
    <button onClick={retry}>Reload</button>
  </div>
)}>
  <MyApp />
</ErrorBoundary>
```

**Two caveats worth knowing:**
1. Error boundaries do **not** catch errors in event handlers, async code (`setTimeout`, promises), or SSR — only render-path errors. Handle those with `try/catch`.
2. If the underlying cause is deterministic (e.g. bad props), clicking retry will just re-throw. For those cases, pair retry with changing the input or add a `resetKeys` prop.
