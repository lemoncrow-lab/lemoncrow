## What it does

- **Catches render errors** — `getDerivedStateFromError` flips the boundary into an error state so React renders the fallback instead of a blank/crashed tree.
- **Fallback UI with retry** — `DefaultFallback` shows the error message plus a **Try again** button. Clicking it calls `reset()`, which clears the error state and re-renders the children (recovers if the error was transient).
- **Logs error details** — `componentDidCatch` calls your `onError` callback and logs the error + `componentStack` to the console.

## Usage

```tsx
import { ErrorBoundary } from './src/ErrorBoundary';

// Basic — default fallback
<ErrorBoundary>
  <MyWidget />
</ErrorBoundary>

// With logging to your telemetry service
<ErrorBoundary onError={(err, info) => Sentry.captureException(err, { extra: info })}>
  <MyWidget />
</ErrorBoundary>

// Custom fallback that reuses the retry button
<ErrorBoundary
  fallback={({ error, retry }) => (
    <div>
      <p>Failed: {error.message}</p>
      <button onClick={retry}>Reload</button>
    </div>
  )}
>
  <MyWidget />
</ErrorBoundary>

// Auto-reset when route/data changes
<ErrorBoundary resetKeys={[userId]}>
  <Profile userId={userId} />
</ErrorBoundary>
```

## Notes worth knowing

- Error boundaries are **class components** by design — React exposes `getDerivedStateFromError`/`componentDidCatch` only to classes, so there's no Hook equivalent. This is expected, not legacy code.
- They catch errors during **render, lifecycle methods, and constructors** of the subtree — **not** in event handlers, async code, or SSR. For those, catch/`setState` manually.
- I swapped in inline styles so the file is drop-in with zero dependencies beyond React; replace `console.error` / the `onError` sink with your real logging pipeline.
