[assistant] # git rebase vs git merge

Both integrate changes from one branch into another. Difference is *how* they do it and what history they leave behind.

## Merge

`git merge feature` creates a new **merge commit** that ties two branch histories together.

```
A---B---C  main
     \   \
      D---E---M  (M = merge commit)
```

- History is preserved exactly as it happened — non-destructive.
- Commit graph shows the branching structure (parallel work is visible).
- Can get noisy: lots of merge commits clutter `git log` on busy repos.

## Rebase

`git rebase main` (run from `feature`) **replays** your commits on top of the target branch, rewriting them as new commits.

```
A---B---C  main
         \
          D'---E'  feature (D,E replayed on C)
```

- Produces a **linear** history — clean, easy to read/bisect.
- **Rewrites commits** (new SHAs). The old commits are discarded.
- No merge commit noise.

## When to use each

**Use merge when:**
- Integrating a finished feature branch into `main` (a `--no-ff` merge documents that a feature landed as a unit).
- The branch is **shared** — others have pulled it.
- You want a truthful record of when/how branches diverged and joined.

**Use rebase when:**
- Cleaning up your **local, unpushed** work before sharing (`git rebase -i` to squash/reorder/fixup).
- Keeping a feature branch up to date with `main` without adding merge commits — replay your work on the latest base.
- You value a linear, bisect-friendly history.

## Tradeoffs / the golden rule

**Never rebase commits that have been pushed and that others may have based work on.** Rebasing rewrites history; collaborators' branches then diverge from yours, causing painful conflicts and duplicated commits. Safe on private branches, dangerous on shared ones.

| | merge | rebase |
|---|---|---|
| History | preserved, branched | rewritten, linear |
| SHAs | unchanged | new |
| Merge commits | yes | no |
| Safe on shared branches | yes | no |
| Conflict resolution | once, at merge | possibly per-commit during replay |

Common hybrid workflow: **rebase locally** to tidy your branch, then **merge** (often `--no-ff` or via PR) into `main`.

One more: `git pull` defaults to merge; `git pull --rebase` rebases your local commits on top of the fetched upstream — handy for keeping a clean linear history while pulling.
