[assistant] # git rebase vs git merge

Both integrate changes from one branch into another, but they build history differently.

## Merge

`git merge` takes the tips of two branches and creates a new **merge commit** that ties them together, preserving both histories exactly as they happened.

```
      A---B---C  feature
     /         \
D---E---F---G---H  main   (H is the merge commit)
```

- History is preserved literally — you can see the branch existed and when it was integrated.
- Non-destructive: existing commits are never changed.
- Can produce a "noisy" graph with many merge commits on busy repos.

## Rebase

`git rebase` **replays** your branch's commits one-by-one on top of another branch, creating new commits with new hashes. The result is a linear history with no merge commit.

```
              A'--B'--C'  feature (replayed onto main's tip)
             /
D---E---F---G  main
```

- History is linear and clean — easy to read with `git log` and `git bisect`.
- Rewrites commits (new SHAs), so it's "destructive" in the sense that the original commits are replaced.
- Resolving conflicts can happen per-commit, which is more tedious than one merge resolution.

## The golden rule

**Never rebase commits that others have already pulled** (i.e., don't rebase shared/public branches). Because rebase rewrites commit hashes, anyone who based work on the old commits will get a divergent history and painful conflicts. Rebase is safe for your **local, unpushed** work.

## When to use each

**Use rebase when:**
- Cleaning up your local feature branch before sharing (`git rebase -i` to squash/reorder/edit commits).
- Keeping a feature branch up to date with `main` while it's still private, so your eventual merge is clean.
- You value a linear, bisect-friendly history.

**Use merge when:**
- Integrating a completed feature into a shared branch like `main` (often via PR).
- The branch is public/shared and rewriting history would disrupt others.
- You want an explicit record that a feature branch was integrated at a point in time.

## Common workflows

- **Rebase then merge:** rebase your feature branch onto the latest `main` locally to linearize, then do a fast-forward or `--no-ff` merge into `main`. Best of both.
- **Merge `main` into feature:** simple and safe for shared feature branches, but clutters history with back-merge commits.
- **`git pull --rebase`:** replays your local commits on top of the fetched upstream instead of creating a merge commit — keeps pull history tidy.

## Quick reference

| | Merge | Rebase |
|---|---|---|
| History | Preserved, branching | Linear, rewritten |
| Creates merge commit | Yes | No |
| Changes commit hashes | No | Yes |
| Safe on shared branches | Yes | No |
| Conflict resolution | Once | Potentially per-commit |

**Rule of thumb:** rebase to clean up *local* work before you share it; merge to integrate *shared* work. When in doubt on a branch others use, merge.
