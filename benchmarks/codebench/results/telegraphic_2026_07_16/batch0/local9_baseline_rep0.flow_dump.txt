[assistant] The race is almost certainly a **read-modify-write** done in your application code. If your handler looks like this:

```js
// ❌ BROKEN: two round-trips, race window between them
const { rows } = await client.query('SELECT value FROM counters WHERE id = $1', [id]);
const next = rows[0].value + 1;
await client.query('UPDATE counters SET value = $1 WHERE id = $2', [next, id]);
return next;
```

Two concurrent requests both read the same `value`, both compute the same `next`, and both write it. They return the same number and you lose an increment.

## Fix: let PostgreSQL do the increment atomically

```js
// ✅ single atomic statement; RETURNING gives you the new value
const { rows } = await client.query(
  'UPDATE counters SET value = value + 1 WHERE id = $1 RETURNING value',
  [id]
);
return rows[0].value;
```

`value = value + 1` is computed inside the database under a row lock, so concurrent requests are serialized on that row and each gets a distinct value. This is the correct fix in the vast majority of cases — no application-level locking needed.

## Why this works

A single `UPDATE` statement is atomic. Postgres takes a row lock on the target row; the second transaction blocks until the first commits, then re-reads and increments from the committed value. No lost updates, no duplicate returns.

## Things that do *not* fix it (common mistakes)

- **Wrapping the SELECT + UPDATE in a `BEGIN/COMMIT`** at the default `READ COMMITTED` isolation level — this alone does *not* help, because the `SELECT` takes no lock and both transactions still read the stale value. You'd need `SELECT ... FOR UPDATE` or `SERIALIZABLE` isolation (with retry-on-serialization-failure logic). The single-statement `UPDATE` is simpler and preferred.
- **A JS mutex / async lock in one Node process** — breaks the moment you run more than one instance or worker.

## If you genuinely need read-then-conditional-write

Use `SELECT ... FOR UPDATE` to lock the row for the duration of the transaction:

```js
await client.query('BEGIN');
const { rows } = await client.query(
  'SELECT value FROM counters WHERE id = $1 FOR UPDATE', [id]
);
const next = rows[0].value + 1;
await client.query('UPDATE counters SET value = $1 WHERE id = $2', [next, id]);
await client.query('COMMIT');
```

Note this must be the **same** `client` (connection) for all three statements — if you're using a `pool`, grab a dedicated connection with `pool.connect()` and `release()` it in a `finally`, otherwise the statements can run on different connections and the lock won't apply.
