**Before split, ask: is monolith actually the problem?** Most "slow" monolith = slow DB queries, missing indexes, N+1, no caching. Microservices fix none of that — add network latency on top. Profile first.

**Key factors:**

**1. Measure real bottleneck.** APM (Django Silk, New Relic, Sentry). Find slow endpoints. Usual suspects:
- N+1 queries — fix with `select_related`/`prefetch_related`
- Missing indexes
- No caching layer (Redis)
- Sync work that should be async (Celery)

Often 80% speedup here, zero architecture change.

**2. Scaling shape.** Microservices help when parts scale differently (one endpoint hammered, rest idle). If whole app scales together, horizontal scaling of monolith (more gunicorn workers/pods behind load balancer) simpler and cheaper.

**3. Team size + ownership.** Microservices = org tool, not perf tool. Pay off when many teams need independent deploy. One small team + microservices = distributed monolith pain (deploy all together anyway, now with network calls).

**4. Data coupling.** Hardest part. Django ORM joins across tables = free. Split services = those joins become API calls + distributed transactions + eventual consistency. If domains tangled, splitting brutal. Map bounded contexts first.

**5. Operational cost.** Each service needs: deploy pipeline, monitoring, logging, service discovery, auth between services, versioning. Distributed tracing now mandatory. Debugging one request spans many logs.

**6. Failure modes.** In-process call never fails mid-way. Network call does — need retries, timeouts, circuit breakers, idempotency.

**Recommended path:**
1. Profile + fix DB/caching. Cheapest, biggest win.
2. Scale monolith horizontally if load-bound.
3. If still need split: **modular monolith** first — clean app boundaries, no cross-app model imports, service-layer interfaces. Gets 80% of decoupling, none of network cost.
4. Extract single loud service only when one part has clear separate scaling/ownership need (e.g. video processing, notifications). Strangler-fig pattern, not big-bang rewrite.

Short: fix perf in monolith now, restructure into modules, extract services one at a time only when real independent-scaling or team need proven.
