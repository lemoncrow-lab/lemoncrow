## The bug: unit mismatch

JWT's `exp` claim is **Unix time in seconds**. `Date.now()` returns **milliseconds**. They differ by a factor of ~1000, so any direct comparison between them is meaningless.

Concretely, right now `Date.now()` ≈ `1.75e12` while a token's `exp` ≈ `1.75e9`. So:

```js
// If your valid-check looks like this:
if (decoded.exp > Date.now()) { /* allow */ }   // 1.75e9 > 1.75e12 → always false
```

That one rejects *everything*. But the version that lets expired tokens through is the mirror image — comparing `exp * 1000` when you think `exp` is something else, or checking expiry in the wrong direction so the condition never fires. Either way the check isn't actually testing what you think.

## The fix

Convert to the same unit. Compare seconds to seconds, or milliseconds to milliseconds:

```js
// Option A: work in seconds
const now = Math.floor(Date.now() / 1000);
if (!decoded.exp || decoded.exp <= now) {
  return res.status(401).json({ error: 'token expired' });
}

// Option B: work in milliseconds
if (!decoded.exp || Date.now() >= decoded.exp * 1000) {
  return res.status(401).json({ error: 'token expired' });
}
```

Note the `!decoded.exp` guard: a token with **no** `exp` claim should be rejected, not treated as never-expiring.

## Better: don't hand-roll it

If you're using `jsonwebtoken`, `jwt.verify()` already validates `exp` for you and throws `TokenExpiredError` — with correct units and optional clock-skew tolerance. Manually decoding and checking `exp` yourself is both the source of this bug and a security risk (you might be reading an *unverified* payload):

```js
const jwt = require('jsonwebtoken');

function auth(req, res, next) {
  const token = (req.headers.authorization || '').replace(/^Bearer\s+/i, '');
  try {
    req.user = jwt.verify(token, process.env.JWT_SECRET); // checks signature AND exp
    next();
  } catch (err) {
    return res.status(401).json({ error: 'invalid or expired token' });
  }
}
```

**Key thing to double-check:** if you were only `jwt.decode()`-ing (not `jwt.verify()`-ing), you weren't validating the signature either — meaning anyone could forge a token with any `exp` they want. Switch to `verify()` and the expiry problem disappears along with a much bigger one.
