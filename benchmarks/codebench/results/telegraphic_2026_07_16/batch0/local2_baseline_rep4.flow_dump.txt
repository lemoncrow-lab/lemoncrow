## What's wrong: seconds vs. milliseconds

A JWT's `exp` claim is a **Unix timestamp in seconds** (per RFC 7519). `Date.now()` returns **milliseconds**. They differ by a factor of 1000:

- `decoded.exp` ≈ `1_752_000_000` (seconds)
- `Date.now()` ≈ `1_752_000_000_000` (milliseconds)

So `decoded.exp` is *always* smaller than `Date.now()`, for every token, forever. Which branch that lands in depends on how you wrote the comparison — and yours is oriented so the "still valid, call `next()`" branch always wins:

```js
// BUG: exp (seconds) is always < Date.now() (ms), so this is always true
if (decoded.exp < Date.now()) {
  next();               // every token passes, including expired ones
} else {
  return res.sendStatus(401);
}
```

The expiry test never actually discriminates — it's a constant. (The mirror-image mistake, `exp > Date.now()` as the valid condition, would reject *everything* instead. Same root cause.)

## The fix

Put both sides in the same units and get the direction right:

```js
const nowSeconds = Math.floor(Date.now() / 1000);
if (decoded.exp < nowSeconds) {
  return res.sendStatus(401);   // expired
}
next();
```

or equivalently `decoded.exp * 1000 < Date.now()`.

## Better: don't hand-roll this at all

If you're using `jsonwebtoken`, `jwt.verify()` already checks `exp` for you (correct units, plus optional clock-skew tolerance) and throws `TokenExpiredError`. A common way to end up with this bug is calling `jwt.decode()` (which does **not** verify anything) and then re-implementing the expiry check yourself:

```js
const jwt = require('jsonwebtoken');

function auth(req, res, next) {
  const token = req.headers.authorization?.split(' ')[1];
  if (!token) return res.sendStatus(401);

  try {
    // verifies signature AND expiry; no manual exp math needed
    req.user = jwt.verify(token, process.env.JWT_SECRET, {
      clockTolerance: 5, // optional: seconds of allowed clock skew
    });
    next();
  } catch (err) {
    // TokenExpiredError, JsonWebTokenError, etc.
    return res.sendStatus(401);
  }
}
```

This also closes a bigger hole: if your current middleware only checks `exp` from a `decode()`d payload, it isn't verifying the **signature** either — so a forged/tampered token would pass. Let `verify()` do both.
