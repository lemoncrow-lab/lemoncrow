[assistant] Here's a solid setup using `node-postgres` (`pg`), which is the standard library for this.

## Installation

```bash
npm install pg
```

## Pool configuration

```javascript
// db.js
const { Pool } = require('pg');

const pool = new Pool({
  host: process.env.PGHOST || 'localhost',
  port: process.env.PGPORT || 5432,
  database: process.env.PGDATABASE,
  user: process.env.PGUSER,
  password: <redacted>,

  // --- Pool sizing ---
  max: 20,                      // max clients in the pool
  min: 2,                       // keep a few warm (pg v8.10+)

  // --- Timeouts (all in milliseconds) ---
  connectionTimeoutMillis: 5000,   // fail if a connection can't be acquired in 5s
  idleTimeoutMillis: 30000,        // close idle clients after 30s
  query_timeout: 10000,            // abort a query that runs longer than 10s
  statement_timeout: 10000,        // server-side cap on statement execution
  maxUses: 7500,                   // recycle a client after N uses (avoids leaks)

  // --- TLS (enable in production) ---
  // ssl: { rejectUnauthorized: true, ca: fs.readFileSync('ca.pem').toString() },
});

module.exports = pool;
```

## Error handling

The most important and most commonly missed piece: **handle errors on idle clients.** Without this listener, a backend error (network drop, DB restart) on an idle client will crash your process.

```javascript
pool.on('error', (err, client) => {
  // Fires when an idle client in the pool emits an error
  console.error('Unexpected error on idle PostgreSQL client', err);
  // Don't exit; the pool will remove the broken client and recover.
});

pool.on('connect', (client) => {
  // Optional: set per-session config for each new connection
  client.query('SET statement_timeout TO 10000');
});
```

## Safe query helper

Wrap queries so clients are **always released**, even when a query throws:

```javascript
async function query(text, params) {
  const start = Date.now();
  try {
    const res = await pool.query(text, params);
    if (Date.now() - start > 1000) {
      console.warn('Slow query', { text, duration: Date.now() - start });
    }
    return res;
  } catch (err) {
    console.error('Query failed', { text, error: err.message });
    throw err;
  }
}

module.exports = { pool, query };
```

For `pool.query()` above, the client is checked out and released automatically. You only need manual release when you check out a client directly (e.g. transactions):

```javascript
async function withTransaction(callback) {
  const client = await pool.connect();
  try {
    await client.query('BEGIN');
    const result = await callback(client);
    await client.query('COMMIT');
    return result;
  } catch (err) {
    await client.query('ROLLBACK');
    throw err;
  } finally {
    client.release();   // ALWAYS release, even on error
  }
}
```

## Graceful shutdown

Drain the pool on shutdown so in-flight queries finish and connections close cleanly:

```javascript
async function shutdown() {
  console.log('Draining PostgreSQL pool...');
  await pool.end();   // waits for active queries, closes all clients
  process.exit(0);
}

process.on('SIGTERM', shutdown);
process.on('SIGINT', shutdown);
```

## Key points on the timeouts

| Option | What it protects against |
|---|---|
| `connectionTimeoutMillis` | Pool exhaustion / DB unreachable — don't hang forever waiting for a client |
| `idleTimeoutMillis` | Wasted connections sitting idle |
| `query_timeout` | Client-side runaway query cancellation |
| `statement_timeout` | Server-side hard cap (survives even if the client hangs) |
| `maxUses` | Slow connection leaks / stale server-side state |

A few practical notes:

- Set **both** `query_timeout` (client-side) and `statement_timeout` (server-side) — they cover different failure modes.
- Create **one pool per process** and reuse it; don't create a pool per request.
- Tune `max` to your DB's `max_connections` divided across all app instances — over-provisioning the pool is a common cause of "too many connections" errors.
