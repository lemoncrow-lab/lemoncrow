[assistant] # git rebase vs git merge

Both integrate changes from one branch into another, but they do it differently and leave different histories.

## What each does

**Merge** creates a new "merge commit" that ties two branch histories together. Your branch's commits stay exactly as they were; a new commit with two parents joins the lines of development.

```
main:    A---B---C
              \       \
feature:       D---E---M   (M = merge commit)
```

**Rebase** re-applies your branch's commits one by one on top of the target branch, creating *new* commits with new hashes. It rewrites history to look linear, as if you'd started your work from the latest tip.

```
main:    A---B---C
                  \
feature:           D'---E'   (D', E' = rewritten commits)
```

## When to use each

**Use merge when:**
- Integrating a finished feature branch into `main` (a merge commit records *when* and *what* was integrated — useful history).
- The branch is shared/public and others have based work on it. Rewriting shared history breaks everyone's copies.
- You want a truthful record of how development actually happened.

**Use rebase when:**
- Updating your *local, private* feature branch onto the latest `main` before opening a PR — keeps history linear and avoids noisy merge commits.
- Cleaning up your own commits before sharing (`git rebase -i` to squash, reorder, reword).
- Your team prefers a linear, bisect-friendly history.

## Tradeoffs

| | Merge | Rebase |
|---|---|---|
| History | Non-linear, preserves true context | Linear, easy to read |
| Commit hashes | Unchanged | Rewritten (new hashes) |
| Safety on shared branches | Safe | Dangerous — never rebase pushed/shared commits |
| Conflict resolution | Once, in the merge commit | Potentially per-commit as each is replayed |
| Traceability | Merge commits can clutter log | Clean, but loses "when merged" context |

## The golden rule

**Never rebase commits that have been pushed and that others may have pulled.** Rewriting shared history forces everyone else into painful recovery. Rebase private work; merge public work.

## Practical workflow many teams use
1. Work on a local feature branch.
2. `git rebase main` periodically to stay current (linear, clean).
3. `git merge --no-ff feature` into `main` when done — a merge commit clearly marks the feature boundary.

A middle-ground option: **squash merge** (common on GitHub PRs) collapses a whole branch into a single commit on `main`, giving linear history without manual rebasing — at the cost of losing individual commit granularity.
