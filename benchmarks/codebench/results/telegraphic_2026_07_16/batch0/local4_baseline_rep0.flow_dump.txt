[assistant] # Git Rebase vs. Git Merge

Both integrate changes from one branch into another, but they do it differently — and the difference shows up in your history.

## What each actually does

**Merge** creates a new "merge commit" that ties two branch histories together. Your feature branch and the target branch both stay intact; the merge commit has two parents.

```
main:    A---B---C-------M
                  \     /
feature:           D---E
```

`M` is the merge commit. History shows *exactly* what happened, including the fact that work happened in parallel.

**Rebase** replays your commits on top of another branch, one by one, creating *new* commits (new SHAs). It rewrites history to look like your work happened sequentially, after the latest changes.

```
main:    A---B---C
                  \
feature:           D'---E'   (D and E replayed as new commits on top of C)
```

There's no merge commit; the history is linear.

## When to use each

**Use merge when:**
- Integrating a completed feature branch into a shared/long-lived branch (`main`, `develop`).
- The branch is public/shared — others have based work on it.
- You *want* the historical record of when and how a branch was integrated.
- You want the safe, non-destructive option.

**Use rebase when:**
- Cleaning up your *own local* commits before sharing them (squash typos, reorder, reword).
- Keeping a feature branch up to date with `main` while you're still working on it, without littering it with merge commits.
- You value a clean, linear, easy-to-`git log` history.

## Tradeoffs

| | Merge | Rebase |
|---|---|---|
| History | Preserves true history, non-linear | Linear, cleaner, but rewritten |
| Commit SHAs | Unchanged | New SHAs (commits rewritten) |
| Traceability | Merge commits show integration points | Lost — looks sequential |
| Conflicts | Resolved once, in the merge commit | May resolve repeatedly (per replayed commit) |
| Safety | Safe on shared branches | Dangerous on shared branches |

## The one rule that matters most

**Don't rebase commits that others have already pulled.** Rebasing rewrites history; if you force-push a rebased shared branch, everyone else's history diverges from yours and you create a mess. The saying is: *rebase locally, merge publicly.*

## A common practical workflow

Many teams combine both:
1. **Rebase** your feature branch onto the latest `main` locally to keep it current and tidy (`git rebase main`).
2. **Merge** it into `main` when done — often with `--no-ff` to preserve a clear integration point, or via a squash-merge on the PR.

This gives you clean local history *and* a meaningful record of when features landed.

A few related tools worth knowing:
- `git rebase -i` (interactive) — squash, reword, reorder, drop commits before sharing.
- `git merge --squash` — collapse a branch into a single commit on merge.
- `git pull --rebase` — rebase your local commits on top of upstream instead of creating a merge commit when pulling.
